## 1. 快速启动

告诉 AI 你的需求，它会自动判断走哪个模式：

| 你的需求 | 自动走 |
|:---------|:-------|
| 「写一本末世小说，5章」 | 长篇管线 |
| 「帮我写一个短篇，离婚反杀爽文，6章」 | 短篇模式 |
| 「继续写上一章」 | session resume（续写模式） |
| 「帮我跑个市场调研，看看末世流最近什么趋势」 | 雷达扫描（Step 1.0） |

你也可以手动指定：

> 「走长篇管线，写一本《XX》」
> 「走短篇模式，都市题材，3章，每章1500字」

## 2. 长篇管线

适合 5 章以上的中长篇。分 4 个 Phase 顺序执行。

### Phase 0：一句话种子

提供故事 premise。例如：

> 「一个在数据废墟中苏醒的 AI，误以为自己是人类考古学家，开始挖掘'人类文明'的遗迹——而它自己就是那个文明最后的遗物。」

AI 会自动补充完整。

### Phase 1：地基构建（Foundation）

产出 5 个文件到 `foundation/` 目录：

| 文件 | 内容 | 必须 |
|:-----|:-----|:----|
| `world.md` | 世界观设定、物理法则、社会结构、历史脉络 | ✅ |
| `characters.md` | 角色档案（核心欲望/恐惧/弧光/声台形表） | ✅ |
| `outline.md` | 章节大纲（三幕结构，每幕约25%/50%/25%） | ✅ |
| `voice.md` | 叙事声音（句长分布/禁忌词/对话标记方式） | ✅ |
| `canon.md` | 硬设数据库 + 伏笔追踪 | ✅ |

**评分门控：** foundation_score ≥ 7.5 才进入 Phase 2。不够就改到够。

**可选 Step 1.0：雷达扫描** — 在动手建世界之前先跑市场调研，看同类作品在干什么。生成 `radar/radar-report.md`。

### Phase 2：初稿（First Draft）

#### Step 2.1：准备

自动创建目录：
```
chapters/       ← 章节文件
runtime/        ← 控制面文件（intent/context/rule-stack/trace）
prompt/         ← 可编辑提示词包（可选，默认空）
materials/      ← 材料归档库（可选，默认空）
```

创建 `state.json`，含字数治理和输入治理字段。

#### Step 2.1b：Plan → Compose（*新融合*）

每章写前自动跑，分为两步：

**1. Plan（规划师）→ `runtime/ch_NN/intent.md`**
- 读 foundation + 上章，生成本章目标
- 输出：must-keep / must-avoid / 可能冲突
- **可编辑**：用户可以直接改 intent.md 调整方向，不需要重新 plan

**2. Compose（编排师）→ 三件套**
- `context.md` — 写手只读这个（精简上下文，800-1200字）
- `rule-stack.yaml` — 规则优先级链（1=章节级 > 5=通用级）
- `trace.json` — 编译轨迹（每个信息从哪里来）

**写手流程：** 读 intent.md（确认方向）→ 读 context.md（上下文）→ 写 ch_NN.md → 字数治理检查

#### Step 2.1d：材料归档检索（*新融合*）

可选功能。在 Step S0 雷达扫描后，或者你有外部参考资料时，可以归档到 `materials/`：

1. 创建 `materials/<分组名>/<文件.md>`
2. 文件头包含元信息（来源/日期/标签）和证据标记（`mat-01-01`）
3. 更新 `materials/index.json`

之后 Compose 在做上下文编译时，会自动检索材料库，把相关条目以摘要形式加入 context.md。

#### 批量委托（30+ 章）

章节数 ≥ 20 时，AI 自动使用 `delegate_task` 并行分发写作，每批 5-11 章。见 Step 2.2a。

### Phase 3：修改循环

#### 3a：自动化修改
- **对抗式编辑**：37维度审计（5组维度，从硬伤到细节）
   - 30+ 章时并行 3 个 worker 分维度审计
- **读者评审**：4 种读者人格并行评审（文学评论家/普通读者/类型爱好者/编辑）
- **修改简报**：汇总 → 优先级标注 → 逐条修复

#### 3b：深度审阅
- 剧情修改确认（先展示计划，用户确认后再改）
- 合并手稿 → 双人格深度审阅 → 逐个修复循环
- 字数治理修复（偏短扩张、偏长压缩）

### Phase 4：导出

- 合并 `manuscript.md`
- 生成 `manuscript-stats.md`（字数/评分/治理通过率）
- 可选：epub 导出、角色关系图谱

## 3. 短篇模式

适合 3-12 章、每章 500-2000 字的独立短篇。比长篇管线快 5-10 倍。

### Step S0：输入方向

一句话 premise。例如：

> 「都市婚姻反转短篇：女主结婚三年发现老公和小三密谋转移财产，她提前拿到账本证据，在离婚协议上反杀。」

AI 会引导补全章节数（默认 6）和每章字数（默认 1200）。

### Step S1：快速地基

只写**必要的**内容：

| 文件 | 内容 | 说明 |
|:-----|:-----|:-----|
| `characters.md` | 核心角色（不要求深层需求/弧光/谎言） | 轻量版 |
| `outline.md` | 每章 1-2 行 | 轻量版 |

**不做** world.md / voice.md / canon.md / 评分门控。

### Step S2：批量撰写

**一次 delegate_task 写完所有章节**——不逐章 Plan→Compose→Draft。短篇每章字数少，逐章跑控制面 overhead 比正文还重。

字数治理短篇宽松版：50-130% target 算合格。

### Step S3：轻量审阅

5 维度快速检查（情节逻辑/角色一致/文笔合规/悬念钩子/整体完成度），最多 1 轮修复。不跑 37 维审计，不跑读者评审。

### Step S4：导出三件套

| 文件 | 内容 | 用途 |
|:-----|:-----|:-----|
| `manuscript.md` | 完整手稿 | 存档 |
| `sales-package.md` | 简介卖点 + 三平台适配推荐语 | **发布** |
| `cover-prompt.md` | 封面提示词（中英文+配色） | **配图** |

## 4. 新增功能详解

### 4.1 输入治理控制面（Plan → Compose）

**作用：** 每章写之前生成方向声明和规则栈，写手只读 compose 产物，不读全量 foundation。

**怎么用：** 不需要手动调用——AI 在每章写前自动跑。

**什么时候手动编辑 intent.md：**

你觉得本章方向不对时（比如「这段对话不该这么写」），直接告诉 AI：

> 「把 runtime/ch_03/intent.md 的 must-avoid 加一条：秦洛不在这章发表长篇大论」

AI 会直接改 intent.md，下一次 compose 自动适配。不需要回退到 plan。

**什么时候查 rule-stack.yaml：**

当你发现规则冲突时（比如 voice.md 要求短句≥40%，但 intent.md 要求本章写长镜头），告诉 AI：

> 「查一下 ch_03 的规则栈，看哪个优先级更高」

AI 会读 rule-stack.yaml 告诉你答案（数字越小越优先）。

### 4.2 可编辑提示词包

**作用：** 外置 agent 提示词为独立文件，用户可直接调整，不动 SKILL.md。

**怎么用：**

在项目 `prompt/` 目录下创建以下任意文件：

| 文件名 | 控制谁 | 示例用途 |
|:-------|:-------|:---------|
| `writer.md` | 写手 agent | 增加「每段不超过3句」 |
| `planner.md` | 规划师 agent | 调整 intent.md 的默认模板 |
| `composer.md` | 编排师 agent | 加入特定排除规则 |
| `auditor.md` | 审计师 agent | 调整审计维度权重 |
| `reviser.md` | 修订师 agent | 约束修改幅度 |

**注意：** 文件存在时覆盖内置默认值。**删除文件**即可恢复默认，不需要重装 skill。

### 4.3 材料归档检索

**作用：** 外部资料（市场报告、年代数据、参考文档）沉淀到项目材料库，不再反复复制粘贴到 prompt。

**怎么用：**

告诉 AI 类似以下指令：

> 「把我之前那份末世小说市场趋势报告存到材料库里」

AI 会自动：
1. 创建或确认分组目录
2. 写入材料文件（含元信息块和证据标记）
3. 更新 `materials/index.json`

之后 Compose 做上下文编译时，自动从材料库检索与本章 POV/场景/标签相关的材料，把摘要加入 context.md。

**适合归档的材料：**
- 雷达扫描报告（`radar/radar-report.md` → 复制到 `materials/市场调研/`）
- 用户提供的参考文档/数据
- 写作中查证的信息（装备参数、年代数据）
- 对抗式编辑中发现的矛盾证据

## 5. 常用指令速查

| 你想做什么 | 说这句 |
|:----------|:-------|
| 写新书 | 「走长篇管线，写一本《XX》」 |
| 继续写 | 「继续」→ AI 会先同步项目状态再问下一步 |
| 跑短篇 | 「走短篇模式，6章，每章1200字，题材是……」 |
| 改方向 | 「把第3章的 must-avoid 加一条：不要写大段心理」 |
| 调提示词 | 「在 prompt/ 下创建一个 writer.md，加一条规则：对话段落不超过3句」 |
| 存材料 | 「把这篇市场报告存到材料库」 |
| 改角色名 | 「把 '王总' 改成 '林北辰'，全局替换」 |
| 审阅 | 「跑一轮对抗式编辑」或「跑深度审阅」 |
| 导出 | 「导出成 manuscript.md」或「导出 epub」 |
| 查状态 | 「现在写到哪了？看看项目状态」 |
| 回滚 | 「把第5章回滚到上一版」 |

## 6. 项目文件结构

```
<项目根目录>/
├── foundation/
│   ├── world.md              世界观设定
│   ├── characters.md          角色档案（含声台形表）
│   ├── outline.md             章节大纲
│   ├── voice.md               叙事声音（含量化风格指纹）
│   ├── canon.md               硬设数据库 + 伏笔追踪
│   └── scores.md              Phase 1 评分
├── radar/                     （可选）雷达扫描报告
├── chapters/                  章节正文
│   ├── ch_01.md
│   └── ...
├── runtime/                   输入治理控制面
│   ├── ch_NN/
│   │   ├── intent.md          ← 可编辑
│   │   ├── context.md
│   │   ├── rule-stack.yaml
│   │   └── trace.json
│   └── character-registry.md
├── prompt/                    （可选）提示词包
│   ├── writer.md / planner.md
│   └── composer.md / auditor.md / reviser.md
├── materials/                 （可选）材料归档库
│   ├── index.json
│   └── 01-分组名/
│       └── 材料文件.md
├── revision/
│   ├── editorial-notes.md
│   ├── reader-reviews.md
│   ├── revision-brief.md
│   ├── deep-review.md
│   ├── fix-log.md
│   └── word-governance-report.md
├── manuscript.md              完整手稿
├── manuscript-stats.md        统计报告
└── state.json                 项目状态
```

**短篇模式的特有文件：**

```
shorts/<短篇名>/
├── foundation/{characters,outline}.md  轻量地基
├── chapters/                            章节正文
├── revision/light-review.md             轻量审阅
├── manuscript.md                        完整手稿
├── sales-package.md                     简介卖点 ← 独有
└── cover-prompt.md                      封面提示词 ← 独有
```
