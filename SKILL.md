# 全自动小说创作工作流（autonovel-workflow）

## 概述

本技能定义了一套完整、可重复的自动化小说创作流水线。灵感来源于 [NousResearch/autonovel](https://github.com/NousResearch/autonovel) 项目的五层协同架构，但完全基于 Hermes Agent 自身的文件操作、delegate_task 子任务分发和评分门控能力实现，**不需要任何外部 Python 脚本或 API Key**。

核心思想：小说创作不是线性填表，而是多层面板同时运转的信息流。世界设定、角色、大纲、叙事声音、实际文本层层耦合，变动双向传播。

## 何时使用

- 你有**一个灵感种子**（一句话 premise/梗概），想扩展成完整小说
- 你有一个自认为不错的故事构思，但不知道如何系统地展开
- 你想要一个**自动化的世界构建 + 角色设计 + 大纲生成**流程
- 你写了一版初稿但**不知道如何修改**，需要对抗式编辑和读者评审
- 你想快速验证一个故事创意的可行性，先跑完地基构建看看评分

## 五层协同架构

每一层解决一个核心问题，从抽象到具体：

```
Layer 5:  voice.md          —— 怎么写（叙事声音、文风调性）
Layer 4:  world.md          —— 什么存在（世界观设定、物理/社会规则）
Layer 3:  characters.md     —— 谁行动（角色档案、动机弧光、关系网）
Layer 2:  outline.md        —— 发生什么（章节大纲、情节 beats）
Layer 1:  chapters/*.md     —— 实际文本（逐章小说正文）
Cross:    canon.md           —— 什么是真的（硬设数据库、不可违背的事实）
```

**变动双向传播原则：**
- **向下影响：** 上层变动（如修改世界观）→ 下层自动调整（如重写相关章节的对话）
- **向上反馈：** 下层写作过程中发现的设定矛盾 → 更新上层 canon.md 和 world.md

## 四阶段流水线总览

```
Phase 1: Foundation  ──→  Phase 2: First Draft  ──→  Phase 3: Revision  ──→  Phase 4: Export
（地基构建）              （初稿撰写）                （修改循环）              （导出成品）
     │                         │                          │                        │
     ▼                         ▼                          ▼                        ▼
 world.md                chapters/NN.md              revision-N.md          manuscript.md
 characters.md           state.json                  reader-review-*.md     word-count stats
 outline.md                                           revision-brief.md
 voice.md
 canon.md
 foundation-scores.md
```

**评分门控：** 只有当前阶段质量达标，才允许进入下一阶段，防止有缺陷的根基影响后续所有工作。

## 前置条件

- **环境：** Hermes Agent 运行环境，`file` 工具集（`write_file`、`read_file`、`patch`）可用
- **Context window：** 建议 ≥32K tokens（长篇复杂小说建议 ≥128K）
- **子任务能力：** `delegate_task` 可用（Phase 3 读者小组评审需要并行分发）
- **工作目录：** 建议为小说创建独立的工作目录 `~/novels/<novel-name>/`，所有文件存放于此
- **可选技能：** 建议使用本技能自带的 `references/anti-slop.md` 和 `references/anti-patterns.md` 进行质量检测，配合 `writing-plans` 技能进行写作规划
- **模板参考：** 本技能自带模板文件在 `templates/` 目录下，可作为 foundation 阶段各文件的起步模板

## Phase 1：地基构建（Foundation）

**目标：** 从种子概念出发，构建完整的五层设定体系。所有产出物存储在 `foundation/` 子目录下。

### Step 1.1：输入种子概念

由用户提供种子概念（一句话 premise）。例如：
> "一个在数据废墟中苏醒的 AI，误以为自己是人类考古学家，开始挖掘'人类文明'的遗迹——而它自己就是那个文明最后的遗物。"

如果用户尚未提供，你可以引导用户想一个。概念应该包含：
- 主角是谁（谁在行动）
- 核心冲突是什么（什么在对抗）
- 设定基调是什么（世界观氛围）

### Step 1.2：生成世界设定（world.md）

创建 `foundation/world.md`，可使用 `templates/world.md` 作为起点。包含以下结构：

```markdown
# 世界设定

## 世界观总览
- 世界名称/时代/类型（科幻/奇幻/现实/混合）
- 一句话摘要

## 物理法则（与真实世界不同的部分）
- 魔法/特异能力系统（如果有）
- 技术/科技水平
- 特殊物质/能量/资源

## 社会结构
- 政治体制
- 社会阶层/种族/阵营
- 经济体系
- 法律与道德规范

## 地理与生态
- 主要地点
- 气候与环境
- 生物/种族分布

## 历史脉络
- 关键历史事件时间线
- 当前故事开始时的时代背景
- 神秘/未知的空白区域

## 基调与氛围
- 核心主题
- 色彩/情绪/感官描述
- 语言风格倾向（宏大/私密/冷峻/温暖）
```

### Step 1.3：生成角色档案（characters.md）

创建 `foundation/characters.md`，可使用 `templates/characters.md` 作为起点。每个核心角色包含：

```markdown
## [角色姓名]

- **身份/标签：** 一句话定位
- **核心欲望：** 最想要的是什么
- **深层需求：** 真正需要的是什么（可能和欲望矛盾）
- **恐惧：** 最怕发生什么
- **谎言/误解：** 自己相信的错误信念
- **人物弧光：** 故事开始时 → 故事结束时 的变化轨迹
- **关键特征：** 外貌、语气习惯、口头禅、行为模式
- **关系网络：** 与其他人物的关系（盟友/对手/爱/恨/矛盾）
- **声音样本：** 一段能体现其说话风格的对白（1-3句）
```

至少包含：主角（protagonist）、对手（antagonist）、主要配角（2-5 人）。

### Step 1.4：生成章节大纲（outline.md）

创建 `foundation/outline.md`，可使用 `templates/outline.md` 作为起点。参考三幕结构或 8-sequence 结构：

```markdown
# 章节大纲

预计总章节数：N

## 第一幕：开端（约25%）
- 第1章：[标题] — 激发事件/日常世界
- 第2章：[标题] — ……
- 第3章：[标题] — 第一扇不可逆之门

## 第二幕：对抗（约50%）
- 第4章：[标题] — 子情节引入
- ……
- 第N章：[标题] — 中点转折（伪胜利/伪失败）
- ……
- 第N章：[标题] — 最后一搏（all is lost moment）

## 第三幕：解决（约25%）
- 第N章：[标题] — 高潮开场
- 第N章：[标题] — 决战/终极冲突
- 最后一章：[标题] — 尾声/新常态

每章大纲格式：
### 第N章：[标题]
- **核心事件：** 发生了什么
- **视角角色：** 从谁的视角叙述
- **关键冲突：** 内部/外部冲突
- **设定揭示：** 本章揭示了什么世界观信息
- **情感节奏：** 开篇氛围 → 中间巅峰情绪 → 收尾调性
- **章节悬念/钩子：** 结尾留下的问题
```

### Step 1.5：生成叙事声音定义（voice.md）

创建 `foundation/voice.md`：

```markdown
# 叙事声音

## 叙述者风格
- 叙事人称（第一/第二/第三人称）
- 叙述者可靠性（可靠/不可靠/有限）
- 叙述者距离（贴近角色/全知/冷眼旁观）

## 语言调色盘
- 句式偏好（长句主导/短句为主/混合）
- 词汇层（口语化/文学化/技术化/诗意）
- 对话标记方式（said-book 式/行动式/无标记）
- 修辞偏好（比喻/隐喻/象征的密度和来源域）

## 节奏标记
- 动作场景的句子长度倾向
- 内心独白的介入频率
- 环境描写的密度和时机

## 禁忌清单
- 该小说绝不会使用的词汇/句式/技巧
- 避免的叙事陷阱（如过度解释/以述代示）
```

### Step 1.6：生成硬设数据库（canon.md）

创建 `foundation/canon.md`，可使用 `templates/canon.md` 作为起点：

```markdown
# 硬设数据库（Canon）

This is what's true. 以下事实不可违背，任何写作中发现矛盾必须更新此文件。

## 不可违背的事实
- （精确的事实列表，如："魔法需要等价交换"、"帝国首都的城墙是黑色的玄武岩"）

## 设定一致性规则
- （如果某设定在其他作品中有不同版本，标明本作品采用的版本）

## 时间线锚点
- （故事中的绝对日期/相对时间）
```

### Step 1.7：评分门控

为 foundation 整体打分。评分维度（每项 1-10）：

| 维度 | 权重 | 问题 |
|:----|:----|:-----|
| 世界观一致性 | 20% | 设定内部有没有矛盾？因果关系是否成立？ |
| 角色深度 | 25% | 每个角色是否有清晰的欲望、恐惧和弧光？ |
| 大纲可行性 | 20% | 章节布局是否合理？节奏分布是否理想？ |
| 声音独特性 | 15% | 叙事声音有没有辨识度？与故事氛围是否匹配？ |
| 灵感潜力 | 20% | 这个设定是否让你迫不及待想写？有足够的延展空间吗？ |

计算公式：`foundation_score = Σ(维度分 × 权重)`

- **≥ 7.5：** 通过！创建 `foundation/scores.md` 记录评分，进入 Phase 2
- **< 7.5：** 退回修改。输出修改建议，回到 Step 1.2-1.6 的对应项修改。最多重试 3 轮，3 轮后仍不达标则建议用户换一个种子概念。

## Phase 2：初稿（First Draft）

**目标：** 逐章撰写初稿。每章一个文件 `chapters/ch_NN.md`，状态记录在 `state.json`。

### Step 2.1：准备工作

```bash
mkdir -p chapters
state.json 内容：
{
  "novel_name": "...",
  "total_chapters": N,
  "current_chapter": 0,
  "completed_chapters": [],
  "last_score": null,
  "status": "writing"
}
```

### Step 2.2：逐章撰写循环

每个章节的撰写流程：

1. **读取上下文：** 读取 `outline.md` 中当前章节的大纲，读取已完成的上一章（如有）
2. **撰写章内：** 遵循 voice.md 的叙事声音，参照 characters.md 的角色特点
3. **自评章节质量（1-10）：** 基于以下维度：
   - 情节推进（本章是否推动了故事）
   - 角色表现（角色行为是否符合设定档案）
   - 文笔流畅度（句子节奏、用词是否自然）
   - 悬念与钩子（结尾是否有读下去的欲望）
4. **门控判断：**
   - **≥ 6.0：** 保留，保存为 `chapters/ch_{NN:02d}.md`
   - **< 6.0：** 标记为需重写，输出具体的修改方向，重新撰写
   - 最多重试 3 次，3 次后仍不合格则接受当前版本并备注
5. **更新 state.json：** 增加 `completed_chapters`，更新 `current_chapter`

### Step 2.3：章节间一致性检查

每完成 3 章，执行一次一致性检查：
- 读取 `canon.md`，检查是否有新增的事实矛盾
- 如果有矛盾，更新 `canon.md`（向上反馈）
- 如果矛盾涉及已完成的章节，在 `patches/` 下记录待修复处（Phase 3 处理）

### Step 2.4：阶段性休息提示

每完成 25% 的章节（即写完整部小说的四分之一），暂停并输出阶段性报告：
- 已写章节数 / 总章节数
- 当前平均分
- 节奏评估（有没有拖沓/过快）
- 建议下一步调整方向

## Phase 3：修改循环（Revision）

**目标：** 对已完成的初稿进行多轮修改，提升质量到可发布水平。

### Phase 3a：自动化修改

按以下顺序执行：

#### Step 3a.1：对抗式编辑

以「最严苛的批评者」身份通读全文，扫描：
- **AI 味检测：** 参考本技能自带的 `references/anti-slop.md` 进行专项检测（词汇级/句式级/结构级检测清单及示例对照）。也可加载 `humanizer` 技能作为补充
- **结构套路检测：** 参考 `references/anti-patterns.md` 检测常见的叙事结构套路
- **冗余检测：** 标记所有可以用一句话说清楚却用了三段的地方
- **过度解释检测：** 标记所有"角色行为 + 作者解释动机"的段落（show, don't tell 违规）
- **情节漏洞：** 标记所有设定不一致或角色行为不合逻辑的地方

输出 `revision/editorial-notes.md`。

#### Step 3a.2：读者小组评审

使用 `delegate_task` 并行分发 4 个 agent，每个扮演一种读者人格：

| 人格 | 任务指令 |
|:----|:--------|
| **文学评论家** | "你是一位严肃的文学评论家。通读以下手稿，评价其文学价值：比喻是否新颖、句式是否多样、主题是否一致、有无明显的公式化处理。给出1-10分和具体改善建议。" |
| **普通读者** | "你只是一个喜欢看故事的路人读者。读完后你的第一反应是什么？哪里觉得无聊？哪里感动了你？哪里让你想弃书？不要用文学术语，用普通人的话说。" |
| **类型文学爱好者** | "你是这个类型（科幻/奇幻/悬疑/言情...）的资深读者。它有没有重复你已经看过一百遍的老套路？有没有新想法？类型 convention 遵守得如何？哪里突破了你预期？" |
| **编辑** | "你是一名专业编辑。评估：市场定位在哪？目标读者是谁？开头前3页是否抓人？哪些段落应该删掉？哪些需要扩展？给出具体的'砍/留/改'建议清单。" |

四个评审意见汇总到 `revision/reader-reviews.md`。

#### Step 3a.3：生成修改简报

综合对抗式编辑结果和读者评审意见，输出 `revision/revision-brief.md`：

```markdown
# 修改简报

## 最高优先级（必须改）
- [ ] 问题1 + 具体章节位置
- [ ] 问题2 + 具体章节位置

## 中等优先级（建议改）
- [ ] 问题3 ...

## 低优先级（可改可不改）
- [ ] 问题4 ...

## 不采纳意见（并说明原因）
- 意见X：不采纳原因
```

#### Step 3a.4：按简报重写

逐条处理修改简报中的条目。每完成一条：
1. 编辑对应章节文件
2. 在简报中标记为 `[x]`
3. 可选的：输出 diff（显示改了什么）

全部完成后，进入 Phase 3b。

### Phase 3b：深度审阅循环

#### Step 3b.1：合并完整手稿

将所有章节按顺序合并为一个文件 `manuscript-draft.md`：

```bash
# 按章节编号从左到右合并
cat chapters/ch_*.md > manuscript-draft.md
```

#### Step 3b.2：双人格深度审阅

一次性以「文学评论家 + 写作教授」双人格审阅整个手稿。

**文学评论家视角：** 哪些段落让你读出声来称赞？哪些段落让你走神？
**写作教授视角（实用）：** 逐章给出建设性批评。关注：
- 开场前 3 句话是否有效
- 每个场景是否有明确的目的
- 对话是否自然、是否推进情节或揭示角色
- 节奏控制：快慢场景是否交替得当
- 情感弧线：读者在每一章体验到的情绪曲线

输出 `revision/deep-review.md`。

#### Step 3b.3：逐个修复循环

1. 按优先级顺序读取 deep-review.md 中的问题
2. 对每个问题进行修复
3. 记录到 `revision/fix-log.md`
4. 全部修复后，**回到 Step 3b.1 重新合并，进入新一轮深度审阅**
5. 当一轮深度审阅中无明显缺陷（标注为「无需修改」的问题 ≥ 90%）时，退出循环

> **注意：** 这个循环可能进行 2-3 轮。不要在这里追求完美——过度修改会杀死作品的生动性。目标是「无明显缺陷」而非「无可挑剔」。

## Phase 4：导出（Export）

### Step 4.1：合并最终手稿

将所有最终版章节合并到一个文件：

```bash
cat chapters/ch_*.md > manuscript.md
```

### Step 4.2：文字统计

输出统计信息保存到 `manuscript-stats.md`：

```markdown
# 手稿统计

## 基本信息
- 小说名称：[名称]
- 总章节数：[N]
- 总字数：[字数]
- 平均每章字数：[平均]
- 最长章节：第N章（字数）
- 最短章节：第N章（字数）

## 质量指标
- Phase 1 地基评分：[分数]
- Phase 2 平均章节得分：[分数]
- Phase 3a 修改条目数：[N]已处理 / [M]总计
- Phase 3b 深度审阅轮次：[N]

## 文件清单
- manuscript.md — 完整手稿
- foundation/world.md — 世界设定
- foundation/characters.md — 角色档案
- foundation/outline.md — 章节大纲
- foundation/voice.md — 叙事声音
- foundation/canon.md — 硬设数据库
- revision/editorial-notes.md — 对抗式编辑记录
- revision/reader-reviews.md — 读者评审记录
- revision/revision-brief.md — 修改简报
- revision/deep-review.md — 深度审阅记录
```

### Step 4.3：可选扩展

根据用户需求，可选操作：
- 导出为 epub（需要 `pandoc` 或 `calibre`）
- 生成封面文案和小说简介
- 生成后续续集/系列扩展建议
- 输出角色关系图谱（Mermaid.js 格式）

```
~/novels/<novel-name>/
```

## 常见陷阱

### 🚫 陷阱 1：地基不够就开写
**表现：** 跳过或不认真执行 Phase 1 的门控评分，直接进入初稿。
**后果：** 写到一半发现设定矛盾/角色平庸/大纲撑不起长篇，需要大改甚至重来。
**解决：** 严格执行 foundation_score ≥ 7.5 门控。如果 3 轮修改后仍不够，诚实地告诉用户这个种子概念可能不成熟。

### 🚫 陷阱 2：评分变成走过场
**表现：** 所有章节都打 6.0+，从不触发重写。
**后果：** 质量低劣的章节累积到 Phase 3 才暴露，修复成本远高于当场重写。
**解决：** 自评时问自己一个关键问题：「如果这是一本你花钱买的书，读到这一章你会不会觉得被坑了？」诚实回答。

### 🚫 陷阱 3：过度修改（over-polishing）
**表现：** 在 Phase 3b 中无限循环，对每个标点符号吹毛求疵。
**后果：** 产出"打磨到死"但失去生气的作品；或干脆永远完不成。
**解决：** 设定退出条件：「无明显缺陷条目 ≥ 90%」。剩下的 10% 可能是个人偏好问题，留给实际读者判断。

### 🚫 陷阱 4：忽视向上反馈
**表现：** 写作中发现设定矛盾，不更新 canon.md 或 world.md。
**后果：** 越往后写矛盾越多，最终 canon 文件失去参考价值。
**解决：** 每完成 3 章强制执行一致性检查。矛盾必须同时修复已写章节和更新 canon。

### 🚫 陷阱 5：声音漂移
**表现：** 第一章和最后一章的文风差异大到像两个人写的。
**后果：** 作品缺乏整体感，读者出戏。
**解决：** 写每章前快速重读 voice.md。如果发现漂移，在 Phase 3a 的统一编辑中纠正。

### 🚫 陷阱 6：章节太短或太长
**表现：** 有的章节 300 字，有的章节 8000 字。
**后果：** 节奏失控，读者体验割裂。
**解决：** 设定每章目标字数范围（建议 1500-3000 字/章）。超出范围的在章节自评阶段标记。

### 🚫 陷阱 7：角色行为不一致
**表现：** 角色在某一章做出了不符合其设定档案的行为。
**后果：** 角色失真，读者失去代入感。
**解决：** 写每章前重读 characters.md 中出场角色的档案。在 Phase 3a 的对抗式编辑中重点检查。
