# 全自动小说创作工作流（autonovel-workflow）

## 概述

本技能定义了一套完整、可重复的自动化小说创作流水线。灵感来源于 [NousResearch/autonovel](https://github.com/NousResearch/autonovel) 项目的五层协同架构，但完全基于 Hermes Agent 自身的文件操作、delegate_task 子任务分发和评分门控能力实现，**不需要任何外部 Python 脚本或 API Key**。

核心思想：小说创作不是线性填表，而是多层面板同时运转的信息流。世界设定、角色、大纲、叙事声音、实际文本层层耦合，变动双向传播。

## 何时使用

- 你有**一个灵感种子**（一句话 premise/梗概），想扩展成完整小说
- 你有一个自认为不错的故事构思，但不知道如何系统地展开
- 你想要一个**自动化的世界构建 + 角色设计 + 大纲生成**流程
- 你写了一版初稿但**不知道如何修改**，需要对抗式编辑和读者评审
- 你想快速验证一个故事创意的可行性，先跑完地基构建看看评分
- 你有一个**已有进度的小说项目**，想「继续」（session resume）——见下方续写模式

## 续写模式（Session Resume）

当用户说「继续」「续写」「接着做」时，不要默认从头开始或猜测上次做到了哪里。执行以下健康检查流程，**先同步项目状态，再呈现选项菜单**。

### Resume Step 1：扫描项目目录

```bash
ls ~/novels/
```

确认项目名称，进入对应目录。如果用户未指定名称，用 `session_search` 查询最近的小说项目会话。

### Resume Step 2：健康检查（文件系统 vs state.json）

逐项检查以下同步状态：

| 检查项 | 方法 | 典型问题 |
|:-------|:-----|:---------|
| state.json 存在且有效 | `read_file('state.json')` | status 字段是否与实际文件匹配 |
| 实际章节文件数 vs completed_chapters | `ls chapters/ch_*.md` 对比 state.json | 可能已创建但未记录，或已记录但文件缺失 |
| 实际中文字数 vs word_governance per_chapter.actual | Python 统计全量章节 | `actual` 可能为 null 或过时 |
| manuscript-stats.md 数据 | 检查总字数、通过率、偏差分布 | 常见于「约XX万字（待填充）」等占位符未更新 |
| revision-brief.md 状态标记 | 检查 ⬜（待处理）是否还有实际已完成的条目 | 修复完成后状态标记未同步 |

**中文字数统计方法（推荐方案）：**

```python
import re
zh_chars = len(re.findall(r'[\u4e00-\u9fff\uf900-\ufaff]', content))
```

> **注意：** 使用 `execute_code` 而非 `terminal` 来运行上述 Python 代码。`terminal` 中复杂的 grep/Python 命令在大文件集上可能超时（blocked），`execute_code` 作为独立执行器更稳定，且不依赖 shell 的 Unicode 处理能力。

### Resume Step 3：修复同步偏差

- state.json 的 `status` 字段与实际阶段不符 → 更新
- 字数统计与实际文件不符 → 用 Python 重新统计并填充
- revision-brief 的 ⬜ → ✅ 按 fix-log.md 的实际记录修复
- manuscript-stats 的占位符 → 填入真实数据

### Resume Step 4：呈现状态摘要 + 选项菜单

```markdown
📋 **《小说名》项目状态**

**已完成：**
- ✅ foundation 5件套
- ✅ 第1-42章大纲
- ✅ 12个补充章节文件
- ✅ 14项修复落地
- ✅ 角色名审查（2处修改）
- ✅ foundation_score 8.2/10

**数据：**
- 总中文字数：约1.16万字（补充章节）
- 补充章节通过率：🟢 16.7%（2/12）
- 42个主章存于大纲，尚未写成独立md文件

**下一步方向：**
① 把主章写成完整md文件
② 扩展偏短章节
③ InkOS字数治理融合
④ InkOS第二阶段（SQLite记忆/Agent拆分）
⑤ 直接导出
```

### Resume Step 5：等待用户选择

不要假设方向。呈现菜单后等待用户指令。如果用户重复说「继续」，列出5个方向让用户选一个数字。


## 五层协同架构

每一层解决一个核心问题，从抽象到具体：

```
Layer 5:  voice.md          —— 怎么写（叙事声音、文风调性）
Layer 4:  world.md          —— 什么存在（世界观设定、物理/社会规则）
Layer 3:  characters.md     —— 谁行动（角色档案、动机弧光、关系网）
Layer 2:  outline.md        —— 发生什么（章节大纲、情节 beats）
Layer 1:  chapters/*.md     —— 实际文本（逐章小说正文）
Cross:    canon.md           —— 什么是真的（硬设数据库、不可违背的事实）
```

**变动双向传播原则：**
- **向下影响：** 上层变动（如修改世界观）→ 下层自动调整（如重写相关章节的对话）
- **向上反馈：** 下层写作过程中发现的设定矛盾 → 更新上层 canon.md 和 world.md

## 四阶段流水线总览

```
Phase 1: Foundation  ──→  Phase 2: First Draft  ──→  Phase 3: Revision  ──→  Phase 4: Export
（地基构建）              （初稿撰写）                （修改循环）              （导出成品）
     │                         │                          │                        │
     ▼                         ▼                          ▼                        ▼
 world.md                chapters/NN.md              revision-N.md          manuscript.md
 characters.md           state.json                  reader-review-*.md     word-count stats
 outline.md                                           revision-brief.md
 voice.md
 canon.md
 foundation-scores.md
```

**评分门控：** 只有当前阶段质量达标，才允许进入下一阶段，防止有缺陷的根基影响后续所有工作。

## 前置条件

- **环境：** Hermes Agent 运行环境，`file` 工具集（`write_file`、`read_file`、`patch`）可用
- **联网：** 如需执行 Step 1.0 雷达扫描，需要 `web_search` 工具可用（非强制——跳过雷达扫描仍可正常执行）
- **Context window：** 建议 ≥32K tokens（长篇复杂小说建议 ≥128K）
- **子任务能力：** `delegate_task` 可用（Phase 3 读者小组评审需要并行分发）
- **工作目录：** 建议为小说创建独立的工作目录 `~/novels/<novel-name>/`，所有文件存放于此
- **可选技能：** 建议使用本技能自带的 `references/anti-slop.md` 和 `references/anti-patterns.md` 进行质量检测，配合 `writing-plans` 技能进行写作规划
- **模板参考：** 本技能自带模板文件在 `templates/` 目录下，可作为 foundation 阶段各文件的起步模板

## Phase 1：地基构建（Foundation）

**目标：** 从种子概念出发，构建完整的五层设定体系。所有产出物存储在 `foundation/` 子目录下。

### Step 1.0：雷达扫描（市场调研）🛰️ *InkOS Radar Agent 融合*

> 在动手构建世界之前，先看看市场上同类作品在干什么。雷达 Agent 不参与写作，只提供市场情报。

**触发条件：** 用户提供了种子概念后，在进入世界构建之前执行。如果用户已有明确的差异化方向，可以跳过。

**执行流程：**

1. **类型定位：** 从种子概念中提取核心标签（如「七零年代文」「先婚后爱」「咸鱼系统」）
2. **市场扫描：** 使用 `web_search` 搜索同类作品，关注：
   - 平台上同类作品的流行元素
   - 常见套路和读者疲劳点
   - 近期的热门差异化方向
3. **竞争分析：** 抽取 2-3 个有代表性的同类作品/热门话题，分析：
   - 它们做得好的地方
   - 被读者吐槽的点
   - 本作品的差异化空间
4. **输出雷达报告：** 保存到 `radar/radar-report.md`

**雷达报告模板：**

```markdown
# 雷达报告 · 《种子概念名称》

## 类型定位
| 标签 | 说明 |
|:----|:-----|
| 主类型 | [年代文/奇幻/言情/...] |
| 子标签 | [七零/先婚后爱/咸鱼/带娃/...] |
| 目标平台 | [起点/晋江/番茄/...] |

## 市场扫描结果
### 同类作品 1：[作品名]
- **人气/评分**：[数据]
- **核心亮点**：
- **读者槽点**：
- **本作品可借鉴**：

### 同类作品 2：[作品名]
- ...

## 差异化机会
| 方向 | 市场现状 | 我们的空间 |
|:----|:---------|:-----------|
| [人设] | 常见套路 | 我们可以反套路做... |
| [系统] | 同质化严重 | 我们创新的点是... |

## 风险预警
- ⚠️ 某个元素可能读者已经看腻了
- ⚠️ 某个设定容易踩雷

## 雷达建议（进入 Phase 1 的输入）
- ✅ 建议在 world.md 中重点突出的差异化方向
- ✅ 建议避开的套路元素
```

> **注意：** 这不是写论文，信息够用就行。3-5 条搜索结果 + 1 个差异化方向足够。关键是让 Planner 知道「别人做了什么、我们不做重复的」。

### Step 1.1：输入种子概念

由用户提供种子概念（一句话 premise）。例如：
> "一个在数据废墟中苏醒的 AI，误以为自己是人类考古学家，开始挖掘'人类文明'的遗迹——而它自己就是那个文明最后的遗物。"

如果用户尚未提供，你可以引导用户想一个。概念应该包含：
- 主角是谁（谁在行动）
- 核心冲突是什么（什么在对抗）
- 设定基调是什么（世界观氛围）

### Step 1.2：生成世界设定（world.md）

创建 `foundation/world.md`，可使用 `templates/world.md` 作为起点。包含以下结构：

```markdown
# 世界设定

## 世界观总览
- 世界名称/时代/类型（科幻/奇幻/现实/混合）
- 一句话摘要

## 物理法则（与真实世界不同的部分）
- 魔法/特异能力系统（如果有）
- 技术/科技水平
- 特殊物质/能量/资源

## 社会结构
- 政治体制
- 社会阶层/种族/阵营
- 经济体系
- 法律与道德规范

## 地理与生态
- 主要地点
- 气候与环境
- 生物/种族分布

## 历史脉络
- 关键历史事件时间线
- 当前故事开始时的时代背景
- 神秘/未知的空白区域

## 基调与氛围
- 核心主题
- 色彩/情绪/感官描述
- 语言风格倾向（宏大/私密/冷峻/温暖）
```

### Step 1.3：生成角色档案（characters.md）

创建 `foundation/characters.md`，可使用 `templates/characters.md` 作为起点。每个核心角色包含：

```markdown
## [角色姓名]

- **身份/标签：** 一句话定位
- **核心欲望：** 最想要的是什么
- **深层需求：** 真正需要的是什么（可能和欲望矛盾）
- **恐惧：** 最怕发生什么
- **谎言/误解：** 自己相信的错误信念
- **人物弧光：** 故事开始时 → 故事结束时 的变化轨迹
- **关键特征：** 外貌、语气习惯、口头禅、行为模式
- **关系网络：** 与其他人物的关系（盟友/对手/爱/恨/矛盾）
- **声音样本：** 一段能体现其说话风格的对白（1-3句）
```

至少包含：主角（protagonist）、对手（antagonist）、主要配角（2-5 人）。

### Step 1.3a：角色自动改名提议 🔄

**目标：** AI 自动检测角色名是否过于普通/常见/同质化，主动提议更名，让用户确认。

**判断标准（触发改名提议的条件）：**

| 问题 | 示例 | 建议动作 |
|------|------|---------|
| 角色名过于常见/套路化 | 「小明」「小红」「王总」「李总」 | 提议更有辨识度的名字 |
| 两个角色名读音或字形相近 | 「张伟」vs「张威」 | 提议拉开差异，避免读者混淆 |
| 角色名与世界观不搭 | 仙侠世界出现「汤姆」「杰瑞」 | 提议符合世界观的东方式/时代名 |
| 名字缺乏特色/记忆点 | 「老王」「小刘」等通用称呼 | 提议赋予特色（如根据角色性格特征命名） |
| 撞名/重名风险 | 与知名作品角色重名 | 提议修改避免侵权或串戏 |

**执行流程：**

```
1. AI 生成初步角色列表（默认名）
2. 对每个角色，AI 自我检查是否触发上述改名条件
3. 列出「原有名 → 建议名」对照表，格式：
   
   | 角色 | 原名 | 建议名 | 理由 |
   |------|------|--------|------|
   | 主角 | 小明 | 林北辰 | 「小明」过俗，「林北辰」更具辨识度和江湖气 |
   | 女主 | 小红 | 苏晚晴 | 「小红」无特色，「苏晚晴」更温婉且有记忆点 |
   
4. 使用 clarify 工具询问用户：
   - 接受所有改名的建议？
   - 部分接受（用户指定哪些改、哪些不改）？
   - 全部保留原名？
5. 根据用户反馈更新 characters.md
6. 根据用户反馈更新 characters.md，然后按 **全局改名执行流程**（见 Step 3a.5 全局改名）落地到所有项目文件。

**注意：** 如果 AI 判断所有角色名都没有触发改名条件（名字已经合理），则跳过此步骤，直接进入 Step 1.4。

### Step 1.4：生成章节大纲（outline.md）

创建 `foundation/outline.md`，可使用 `templates/outline.md` 作为起点。参考三幕结构或 8-sequence 结构：

```markdown
# 章节大纲

预计总章节数：N

## 第一幕：开端（约25%）
- 第1章：[标题] — 激发事件/日常世界
- 第2章：[标题] — ……
- 第3章：[标题] — 第一扇不可逆之门

## 第二幕：对抗（约50%）
- 第4章：[标题] — 子情节引入
- ……
- 第N章：[标题] — 中点转折（伪胜利/伪失败）
- ……
- 第N章：[标题] — 最后一搏（all is lost moment）

## 第三幕：解决（约25%）
- 第N章：[标题] — 高潮开场
- 第N章：[标题] — 决战/终极冲突
- 最后一章：[标题] — 尾声/新常态

每章大纲格式：
### 第N章：[标题]
- **核心事件：** 发生了什么
- **视角角色：** 从谁的视角叙述
- **关键冲突：** 内部/外部冲突
- **设定揭示：** 本章揭示了什么世界观信息
- **情感节奏：** 开篇氛围 → 中间巅峰情绪 → 收尾调性
- **章节悬念/钩子：** 结尾留下的问题
```

### Step 1.5：生成叙事声音定义（voice.md）

创建 `foundation/voice.md`（InkOS 融合模式——定性+定量双轨）：

```markdown
# 叙事声音

## 一、量化风格指纹（InkOS 风格分析）
*参考自 InkOS `inkos style analyze` 的风格指纹提取法*

### 1.1 句长分布
- 平均句长：[N] 字
- 中位句长：[N] 字
- 分布区间：短句（≤15字）占比、中句（16-30字）占比、长句（31-50字）占比、超长句（>50字）占比
- **规则：** 核心节奏比例——短句主导还是中句主导？

### 1.2 段落密度
- 平均每章段数
- 每段字数区间
- 段落切分偏好（快切式/绵延式/混合）

### 1.3 对话占比
- 对话占全书字数比例
- 对话标记方式（行动式/said-book/无标记）

### 1.4 高频字/词指纹
- 高频 TOP10（用于漂移检测）
- 避免使用的词汇清单（检测漂移的风向标）
- 风格辨识性特征

## 二、叙述者风格
- 叙事人称（第一/第二/第三人称）
- 叙述者可靠性（可靠/不可靠/有限）
- 叙述者距离（贴近角色/全知/冷眼旁观）
- POV 切换规则

## 三、语言调色盘
- 句式偏好（长句主导/短句为主/混合）
- 词汇层（口语化/文学化/技术化/诗意）
- 对话标记方式（said-book 式/行动式/无标记）
- 修辞偏好（比喻/隐喻/象征的密度和来源域）

## 四、节奏标记
- 动作场景的句子长度倾向
- 内心独白的介入频率
- 环境描写的密度和时机
- 不同场景类型的节奏指南

## 五、禁忌清单
- 该小说绝不会使用的词汇/句式/技巧
- 避免的叙事陷阱（如过度解释/以述代示）

## 六、参考案例（可选）
- 已通过验证的样本章节链接（写前快速翻读校准语调）
```

**技巧：** 如果项目已有初稿章节，运行以下脚本提取实际量化数据：

python3 ~/.hermes/skills/creative/autonovel-workflow/scripts/style-fingerprint.py --dir chapters/
```

脚本输出会分析句长分布、段落密度、对话占比、高频字和违禁词，直接填入 voice.md。无初稿则设定目标值。关键不是数字精确，而是有个可量化的参考锚点防止声音漂移。

> 🔄 InkOS `inkos style analyze` 模式：如果分析工具可用，运行 `inkos style analyze <chapter.md>` 提取指纹后直接注入 voice.md。不可用时，手动提取 3-5 章典型样本即可建立基准。

### Step 1.6：生成硬设数据库（canon.md）

创建 `foundation/canon.md`，可使用 `templates/canon.md` 作为起点：

```markdown
# 硬设数据库（Canon）

This is what's true. 以下事实不可违背，任何写作中发现矛盾必须更新此文件。

## 不可违背的事实
- （精确的事实列表，如："魔法需要等价交换"、"帝国首都的城墙是黑色的玄武岩"）

## 设定一致性规则
- （如果某设定在其他作品中有不同版本，标明本作品采用的版本）

## 时间线锚点
- （故事中的绝对日期/相对时间）

## 伏笔追踪（Pending Hooks）🔗 *参考 InkOS 的 foreshadowing retrieval 维度*
*此表是写作/编辑过程中持续维护的活文档，Phase 3a V21（伏笔回收率）检测时以此为据。*

| Status | 含义 |
|:------:|:-----|
| 🟢 已回收 | 伏笔已揭示/解决，回收位置确认 |
| 🟡 待回收 | 伏笔已埋设，预期回收章节未到 |
| 🔴 逾期未收 | 预期回收窗口已过但未回收（须在 Phase 3 处理） |
| ⚪ 艺术留白 | 故意不回收的开放式伏笔（需在备注说明理由） |

### 主线伏笔
| # | 伏笔内容 | 埋设位置 | 预期回收 | 实际回收 | 状态 | 备注 |
|:-:|:---------|:---------|:---------|:---------|:----|:-----|

### 情感/角色弧光伏笔
同上结构。

### 维护说明
1. Phase 2 每写一章，检查是否有新埋伏笔 → 追加到表（状态=🟡）
2. 每完成一章，检查是否回收了任何 🟡 条目 → 更新为 🟢
3. V21 计算：回收率 = 🟢 / (🟢+🟡+🔴)，目标 ≥ 90%
4. 艺术留白必须注明理由，防误报
```

### Step 1.7：评分门控

为 foundation 整体打分。评分维度（每项 1-10）：

| 维度 | 权重 | 问题 |
|:----|:----|:-----|
| 世界观一致性 | 20% | 设定内部有没有矛盾？因果关系是否成立？ |
| 角色深度 | 25% | 每个角色是否有清晰的欲望、恐惧和弧光？ |
| 大纲可行性 | 20% | 章节布局是否合理？节奏分布是否理想？ |
| 声音独特性 | 15% | 叙事声音有没有辨识度？与故事氛围是否匹配？ |
| 灵感潜力 | 20% | 这个设定是否让你迫不及待想写？有足够的延展空间吗？ |

计算公式：`foundation_score = Σ(维度分 × 权重)`

- **≥ 7.5：** 通过！创建 `foundation/scores.md` 记录评分，进入 Phase 2
- **< 7.5：** 退回修改。输出修改建议，回到 Step 1.2-1.6 的对应项修改。最多重试 3 轮，3 轮后仍不达标则建议用户换一个种子概念。

mkdir -p chapters

**创建 `state.json`，包含字数治理和输入治理字段：**

```json

  "novel_name": "...",

  "current_chapter": 0,

**字数目标确定规则：**

- **默认值：** 2500字/章（基于中文网文读者阅读习惯，10-15分钟/章）
- **特殊章节覆盖：** 在 outline.md 中标注 `**字数目标**: NNNN` 的，按标注值覆盖
- **场景类型参考：**

| 场景类型 | 建议字数 | 说明 |
|:---------|:--------:|:-----|
| 日常/建立 | 2000-2500 | 铺垫信息，不需要太密 |
| 对话/情感 | 2500-3500 | 对话场景需要空间展开 |
| 冲突/转折 | 3000-4000 | 高潮需要足够的篇幅制造节奏 |
| 过渡/喘息 | 1500-2000 | 缓冲区，写太满会疲劳 |
| 章节精校 | 按目标 ±20% | Phase 3b 治理修复 |

### Step 2.1b：规划师+编排师管线 (Plan → Compose) 🧠 *InkOS Input Governance 融合*

> 在写手动笔之前，分两步编译控制面输入：
> 1. **规划师 (Plan)** — 读 foundation + 上章 + 状态，生成本章意图声明（intent.md）
> 2. **编排师 (Compose)** — 读 intent + foundation + 状态，编译上下文包（context.md）+ 规则栈（rule-stack.yaml）+ 编译轨迹（trace.json）
>
> 写手只需读 compose 产物，不再读全量 foundation 文件。控制面文件可审核、可编辑、不混进 LLM prompt。

**执行时机：** 每章撰写前执行。第一章做一次；后续章节在上章完成后、下章开始前执行 plan→compose 两步。

---

#### Step 2.1b-1：规划师意图声明 (Plan)

**输入：** `foundation/*.md` + `state.json` + `chapters/ch_{NN-1}.md`（上章结尾）+ 当前 `foundation/outline.md` 中的本章大纲

**输出：** `runtime/ch_{NN}/intent.md`（本章意图声明 ≈ 300-500 字）

**执行流程：**

1. **读作者意图：** 提取 `foundation/voice.md` 的核心风格规则、`foundation/canon.md` 的活跃硬设
2. **读本章大纲：** 从 `foundation/outline.md` 提取本章的 POV、核心事件、悬念、字数目标
3. **读上章结尾：** 上章末段（时间/地点/未解决事件）
4. **编译 intent.md：** 使用模板 `templates/chapter-intent.md`

**intent.md 模板：**

```markdown
# 第{NN}章意图声明

## 本章目标
- {一句话描述本章在故事中的功能}

## 必须保留 (must keep)
- {POV：不得偏离的视角}
- {核心事件：必须推进的情节}
- {基调：必须维持的情感氛围}

## 必须避免 (must avoid)
- {硬设违背：不能破坏 canon.md 中的某项规则}
- {声音漂移：不能使用某类词汇/句式}
- {节奏陷阱：不能写太密/太散}

## 可能冲突
- {如果 A 发生，B 设定可能矛盾 → 处理方式}
```

> **意图声明不用于生成正文，只用于方向对齐。** Compose 基于它选择上下文，Writer 基于它确认方向。用户或 AI 可以随时编辑 intent.md 来微调方向，不需要重跑 plan。

---

#### Step 2.1b-2：编排师上下文编译 (Compose)

**输入：** `runtime/ch_{NN}/intent.md` + `foundation/*.md` + `state.json` + `chapters/ch_{NN-1}.md`（上一章）

**输出：** `runtime/ch_{NN}/` 下的三件套：
- `context.md` — 精简上下文（≈ 800-1200 字），供 Writer 直接阅读
- `rule-stack.yaml` — 规则优先级链（章节级 > 卷级 > 书级 > 通用）
- `trace.json` — 输入编译轨迹（哪个文件读了哪些内容、为何选入/排除）

**编译流程（纯文件读取+信息提取，不调用 LLM）：**

1. **读 intent.md：** 提取 must-keep / must-avoid / conflicts
2. **读 outline.md：** 提取本章大纲（POV、核心事件、悬念、字数目标）
3. **读 voice.md：** 提取量化风格指纹（句长目标、禁忌词）和禁忌清单
4. **读 canon.md：** 提取与本章 POV/场景相关的「不可违背的事实」，提取 pending_hooks 中状态 🟡/🔴 的伏笔
5. **读上一章（如有）：** 提取结尾 1-2 段（衔接点）
6. **读 characters.md：** 提取本章出场角色的当前状态和行为基线
7. **编译 context.md：** 使用 `templates/context-pack.md` 模板填充
8. **编译 rule-stack.yaml：** 按优先级声明规则来源
9. **编译 trace.json：** 记录每个信息的来源文件和选择理由

**context.md 结构和内容量控制：**

```markdown
# 上下文 · 第N章

> 由编排师自动编译。来源：intent.md + outline + voice + canon + 上章

## 章节基本信息
| POV | 字数目标 | 场景类型 |
|:----|:--------:|:--------:|

## 本章意图概要
- 核心任务：{从 intent.md 提取}
- must-keep：{1-2 条最关键的}
- must-avoid：{1-2 条最关键的}

## 活跃硬设（仅本章相关）
- （从 canon.md 中滤取，不超过 5 条）

## 活跃声音规则
- 句长：短句 ≥ X%
- 禁用词：禁止 X、Y、Z
- （从 voice.md 中滤取，不超过 5 条）

## 本场角色
| 角色 | 入章状态 | 行为禁忌 | 声台形表锚点 |
|:----|:---------|:---------|:------------|

## 与上章衔接
- 时间/地点
- 未解决事件（上章结尾钩子）

## must do / must avoid
### ✅ 本章必须推进
### ❌ 本章禁止出现
```

**rule-stack.yaml 格式**（规则优先级：数字越小优先级越高）：

```yaml
# 规则栈 - 第NN章
# 优先级: 1=最高(章节级), 5=最低(通用)
rules:
  - priority: 1
    source: "intent.md"
    scope: "chapter"
    rule: "必须回避 POV_A → 用 POV_B 推进"
  - priority: 2
    source: "outline.md"
    scope: "chapter"
    rule: "本章核心事件：XXXX"
  - priority: 3
    source: "canon.md:不可违背的事实"
    scope: "chapter"
    rule: "魔法需要等价交换"
  - priority: 4
    source: "voice.md:量化风格指纹"
    scope: "book"
    rule: "短句占比 ≥ 40%"
  - priority: 5
    source: "autonovel-workflow:通用规则"
    scope: "global"
    rule: "禁用词：忽然、好像、忍不住、突然、猛地、却"
```

**trace.json 格式**（输入编译轨迹，用于调试和回溯）：

```json
{
  "chapter": "NN",
  "composed_at": "2026-07-07T12:00:00Z",
  "sources": [
    {"file": "runtime/ch_NN/intent.md", "purpose": "意图方向", "selected": true},
    {"file": "foundation/outline.md", "purpose": "POV+核心事件", "selected": true},
    {"file": "foundation/voice.md", "purpose": "风格指纹+禁忌", "selected": true},
    {"file": "foundation/canon.md", "extracted": ["不可违背事实 3条", "pending_hooks 2条"], "selected": true},
    {"file": "foundation/characters.md", "extracted": ["角色A状态", "角色B状态"], "selected": true},
    {"file": "chapters/ch_XX.md", "purpose": "上章衔接", "selected": true}
  ],
  "excluded": [
    {"file": "foundation/world.md", "reason": "本章不涉及新地点/生态"}
  ]
}
```

> **直接写！** context.md 不追求完美排版。关键是**让写手在 30 秒内知道自己要写什么、有什么限制**。信息精确优先，格式次之。

> **rule-stack.yaml 不用于生成正文**，只作为审阅和调试的参考。当 Writer 或 Auditor 发现规则冲突时，按优先级数字解决——数字越小越优先。

> **trace.json 不用于写作**，只在 Step 3a 对抗式编辑和排错时用于回答「为什么本章有/没有某个信息」。

> **缓存失效规则：** foundation 文件在上次编译后有修改（如 canon.md 新增设定），state.json 标记 `canon_stale: true`，编排师在下一次编译时重新读取。intent.md 被用户手动修改后，下一次 compose 直接读取最新 intent 而不需要重新 plan。

#### Step 2.1b-3：控制面文件维护规则

| 文件 | 谁维护 | 何时更新 | 可否人工编辑 |
|:----|:-------|:---------|:-----------|
| `intent.md` | 规划师自动生成 | 每章 plan 时 | ✅ 可编辑，修改后 compose 自动使用最新版 |
| `context.md` | 编排师自动编译 | 每章 compose 时 | ❌ 不直接编辑——改 intent.md 后重跑 compose |
| `rule-stack.yaml` | 编排师自动编译 | 每章 compose 时 | ❌ 不直接编辑 |

### Step 2.1c：可编辑提示词包 📝 *InkOS Editable Prompt Packs 融合*

> 将写手/审计师/编排师等 agent 的核心提示词外置为独立文件，用户可直接编辑调整。
> 不必修改 SKILL.md，也不影响其他项目。

#### 提示词包结构

```
prompt/                         # 项目级提示词包覆盖目录（可选）
├── writer.md                   # 写手提示词——章节撰写指令
├── planner.md                  # 规划师提示词——intent.md 生成指令
├── composer.md                 # 编排师提示词——上下文编译指令
├── auditor.md                  # 审计师提示词——37维度审计指令
└── reviser.md                  # 修订师提示词——修复执行指令
```

每个文件对应一个 agent 的核心 prompt。如果文件存在，agent 在运行时使用该文件内容替代内置默认提示词；如果不存在，使用 SKILL.md 中内置的默认值。

#### 加载规则（三层层叠）

| 优先级 | 来源 | 示例 |
|:------|:-----|:-----|
| 1（最高） | `prompt/<agent>.md` | `prompt/writer.md` |
| 2 | 项目 `.inkos/prompts/`（InkOS 兼容） | 跨项目共享时使用 |
| 3（内置默认） | SKILL.md 中内置的系统提示 | 始终存在作为 fallback |

**查找顺序：** 先查 `prompt/<agent>.md` → 查到就用 → 查不到用内置默认。

#### 何时编辑提示词包

| 场景 | 建议编辑的文件 | 例子 |
|:-----|:-------------|:-----|
| 写手风格需要强约束 | `prompt/writer.md` | 增加「每段不超过3句」规则 |
| 审计维度需要加权重 | `prompt/auditor.md` | 把「伏笔回收率」权重从5%提到15% |
| 编排师需要排除某类文件 | `prompt/composer.md` | 让编排师不加载 `foundation/world.md` 中的地理部分 |
| 规划师意图生成方向偏了 | `prompt/planner.md` | 调整「必须避免」的默认模板结构 |

#### 编辑原则

1. **不覆盖核心逻辑** — 只微调语气/侧重/约束，不改 agent 的行为流程（如「先读intent再读outline」这种流程不放在prompt里）
2. **保持 diff 可追踪** — 只在文件头部添加注释说明改了什么、为什么改。例如：
   ```markdown
   <!-- 2026-07-07 枫火：增加段落长度限制，因为对话场景太多长段落 -->
   - 每段不超过5句
   - 对话段落不超过3句
   ```
3. **可以回退** — 删除 prompt 目录下的对应文件即可恢复内置默认值，不需要重装技能

#### 示例：写手提示词包（prompt/writer.md）

```markdown
# Writer Prompt — 写手提示词
# 当存在此文件时，写手使用以下规则替代内置默认值。
# 删除此文件即可恢复内置默认。

## 风格约束
- 每段不超过5句
- 对话段落不超过3句
- 动作场景以短句（≤12字）为主

## 禁止行为
- 不要在段落开头重复上一段末句信息
- 不要使用「就这样」「于是」「接下来」等转场词
- 角色对话中不要出现说明性长篇独白

## 字数提示
目标字数在 context.md 的「章节基本信息」表中。
偏短勿长：字数不够可以在 Phase 3 扩张，写长了不好砍。
```

> **默认不创建这些文件。** 提示词包是**按需添加**的——只有当用户对某个 agent 的输出方向有明确调整需求时才创建对应文件。项目初始化时 `prompt/` 目录为空，所有 agent 使用内置默认值。

### Step 2.2：逐章撰写循环

每个章节的撰写流程：

1. **读取意图声明：** 快速浏览 `runtime/ch_{NN}/intent.md`，确认本章目标、must-keep、must-avoid。如需微调方向，直接编辑 intent.md 后重跑 compose（不需要回退到 plan）。
2. **读取上下文包：** 读取 `runtime/ch_{NN}/context.md`。这就是本章需要的全部上下文，包含 POV、核心事件、意图概要、活跃硬设、声音规则、角色状态、上章衔接。**不需要再读 foundation/*.md 的内容**
3. **读取规则栈（可选）：** 如果写作中出现规则冲突（如声音规则 vs 场景需求），查阅 `runtime/ch_{NN}/rule-stack.yaml` 按优先级解决——数字越小越优先。
4. **读取字数目标：** 从 `intent.md`（优先）或 `state.json` 的 `word_governance.per_chapter` 中读取当前章节的 `target` 值
5. **撰写章节：** 遵循 voice.md 的叙事声音，参照 characters.md 的角色特点，遵循 intent.md 的 must-keep/must-avoid 方向。当 intent.md 和 context.md 出现矛盾时，以 intent.md（规划师方向）为准。
6. **字数治理检查（InkOS 保守写模式）：**
   计算章节实际字数（去空白后统计 `zh_chars` 中文字符数），与 `target` 对比：

   | 偏差 | 处理 | 说明 |
   |:----|:-----|:-----|
   | **< 50% target** | 🔴 **严重不足** → 暂不保存，必须在当前轮次补写到 ≥ 60% | 明显信息量不够，趁热补 |
   | **50-70% target** | 🟡 **偏少** → 保存但标记 `ok:"short"`，Phase 3 扩张 | 保守写模式下可接受 |
   | **70-130% target** | 🟢 **合格** → 保存，标记 `ok:true` | 目标区间内 |
   | **> 130% target** | 🟡 **超长警告** → 保存但标记 `ok:"overshoot"`，Phase 3 压缩 | 长章修改成本高，须在 Phase 3 处理 |
   | **> 200% target** | 🔴 **严重超长** → 发出强警告，建议立即拆分或大幅压缩 | 写完了但必须大改 |

   > **`--words <N>` 覆盖：** 如果本章是高潮/名场面，可以在撰写前声明 `--words 4000` 覆盖默认 target，避免误报。覆盖值写入 state.json 的 `override` 字段。
7. **自评章节质量（1-10）：** 基于以下维度：
   - 情节推进（本章是否推动了故事）
   - 角色表现（角色行为是否符合设定档案）
   - 文笔流畅度（句子节奏、用词是否自然）
   - 悬念与钩子（结尾是否有读下去的欲望）
8. **门控判断：**
   - **≥ 6.0：** 保留，保存为 `chapters/ch_{NN:02d}.md`
   - **< 6.0：** 标记为需重写，输出具体的修改方向，重新撰写
   - 最多重试 3 次，3 次后仍不合格则接受当前版本并备注
9. **更新 state.json：** 检查 `completed_chapters`，更新 `current_chapter`。同时更新 `word_governance.per_chapter.ch_NN.actual` 为实际字数、`ok` 为治理结果。

### Step 2.2a：批量委托策略（30+ 章长篇小说专用）🤝 *delegate_task 并行分发*

> 当小说篇幅超过 30 章时，逐章写作在单个会话中不可行（每章约 2-5 分钟，54 章就要 2-4 小时）。**使用 `delegate_task` 并行分发章节写作**，将 50+ 章在 2 轮内完成。

**何时使用：** 总章节数 ≥ 20 章时，优先考虑批量委托而非逐章撰写。20 章以下仍可单章写。

**执行流程：**

1. **分片规划：** 按章节跨度合理分片，每片 5-11 章。参考模式：
   - 小说 54 章 → 分 5 片（约 11+9+15+6+13 章），2 轮委托（每轮 2-3 个 `delegate_task` 并发）
   - 每片应是一个**叙事单元**（如「穿越换亲篇」「创业商战篇」），避免在长对话场景中间切分

2. **构造上下文包：** 每个子任务需要以下上下文：
   - 项目路径和文件名命名规范
   - `voice.md` 关键规则（短句比例、禁用词、对话标记方式、加粗金句限制）
   - `canon.md` 核心设定（系统规则、不可违背事实、年代经济参数）
   - 该批次的每章大纲：章节号、POV、核心事件、名场面、情感节点
   - **控制面输入（可选但推荐）：** 如果该项目已有 `runtime/ch_NN/intent.md`（规划师意图声明），将其摘录给子任务而非让子任务自行编译——减少 token 消耗的同时保持方向一致
   - 明确标注哪些章节已有文件，不要覆盖

3. **设置目标：** 每个子任务写明「需要写的文件：ch_XX.md ~ ch_YY.md，共 N 章」。**不要**一次性给子任务 15+ 章——它会在 600s 超时前勉强写完但无法汇报。

4. **处理超时：** `delegate_task` 内置 600 秒超时。**实测发现：超时的子任务文件通常已经写入磁盘**——超时发生在上传汇报阶段，而非写作阶段。收到超时错误后，用 `ls chapters/` 确认文件是否已创建。如果文件存在（通常如此），无需重试。

5. **批次划分参考（54 章实测）：**

| 批次 | 章节范围 | 新写 | 已有 | 结果 |
|:----|:---------|:---:|:---:|:----:|
| A | 第3-13章（穿越换亲篇） | 10 | 1 | ⏱️ 超时但文件全写完了 |
| B | 第14-22章（婚姻系统篇） | 8 | 1 | ✅ 正常完成 |
| C | 第23-37章（创业商战篇） | 10 | 5 | ✅ 正常完成 |
| D | 第38-46章（商战后半） | 6 | 0 | ⏱️ 超时但文件全写完了 |
| E | 第47-54章（结局篇） | 7 | 0 | ⏱️ 超时但文件全写完了 |

6. **后处理：** 所有子任务完成后，用 `ls chapters/ch_*.md | wc -l` 核对文件总数，确认没有缺失。

#### Step 2.2a.1：后处理——批量验证清单

> 子任务写完后不要直接跳过。用 `execute_code` 运行以下验证脚本，对**本次补写的所有章节文件**逐一检查合规性。发现违规立即 `patch` 修正。

**验证检查项（优先级从高到低）：**

| 检查项 | 方法 | 违规处理 |
|:-------|:-----|:---------|
| ① **禁用词扫描** | 统计 voice.md 中列出的禁用词（忽然/好像/忍不住/突然/猛地/然而/但是/却 等）在每个新文件中的出现次数 | 对每个违规实例 `patch` 替换（直接删除或替换为等义短句）。常见替换模式：「忽然」→ 删除；「猛地」→ 删除；「忍不住」→ 「没忍住」或删除；「却」→ 用断句替代 |
| ② **加粗计数** | 统计 `**...**` 在每章中的出现次数。仅排除结构标记 `**POV**` | 如果某章内容加粗 >1 处，取消过度加粗，保留该章最有力的 1 处作为名场面标记 |
| ③ **字数范围** | `zh_chars = len(re.findall(r'[\u4e00-\u9fff\uf900-\ufaff]', content))` | 2000-3500 字为理想区间。低于 1500 → 需要确认是否信息量不足。超过 4000 → 检查是否跑题或混入了多个场景 |
| ④ **「看着XXX」句式** | 搜索 `看着` 开头的中文短句 | 替换为动作式叙述（「她看着他的背影」→「他转身走了。她站在原地没动。」）|

**验证脚本模板：**

```python
import re, os

base = '<project_path>/chapters'
targets = ['ch_XX.md', 'ch_YY.md', ...]  # 本次补写的文件
forbidden = ['忽然','好像','忍不住','突然','猛地','然而','但是','却']

for fname in targets:
    path = os.path.join(base, fname)
    with open(path) as f:
        content = f.read()
    zh = len(re.findall(r'[\u4e00-\u9fff\uf900-\ufaff]', content))
    bolds = [b for b in re.findall(r'\*\*[^*]+\*\*', content) if b != '**POV**']
    fb = {w: content.count(w) for w in forbidden if content.count(w) > 0}
    print(f'{fname}: {zh}字, 加粗={len(bolds)}, 禁用词={fb}')
```

**注意：** 使用 `execute_code` 而非 `terminal`，避免 shell 对 Unicode 的截断。验证通过后，将本次补写新增到 state.json 的 `completed_chapters` 和 `word_governance.per_chapter` 中。

**注意事项和坑：**

- ⚠️ **不要在 `delegate_task` 的 context 中放完整的大纲文件。** 只放该批次相关的大纲摘录——否则 token 会爆炸，子任务响应变慢甚至超时
- ⚠️ **每个子任务请明确标注跳过哪些已有文件**（如「第5章已有 ch_04b.md，跳过」「第22章已有 ch_20b.md，跳过」）
- ⚠️ **不要给子任务 15+ 章。** 10 章以内是比较稳妥的上限。超过这个数，子任务可能在前几章消耗过多 token，后面急赶质量下降
- ✅ **子任务中嵌入风格指南**（voice.md 的关键规则）比让子任务自行 read_file 更可靠——read_file 可能返回截断内容
- ✅ **文件命名分离：** 子任务按新编号写 `ch_NN.md`，与已有的 b/c 后缀文件（如 `ch_04b.md`）不会冲突

### Step 2.2b：观察者事实提取 🔍 *InkOS Observer Agent 融合*

> 每一章写完后，观察者自动提取本章中产生的**新事实**，更新到 canon.md 和状态文件。
> 这是五层架构「向上反馈」的具体实现——写作中发现的设定变化，不应等 3 章后才处理。

**执行时机：** 每一章完成并保存后，立即执行。是 Step 2.2 → Step 2.2b → Step 2.3 的顺序。

**输入：** 刚完成的 `chapters/ch_{NN}.md` + 现有的 `foundation/canon.md`

**提取的 9 类事实（InkOS v1.6+ 扩展）：**

| 类别 | 提取内容 | 更新目标 |
|:----|:---------|:---------|
| 🆕 **新角色** | 本章第一次出现的有名角色 | `runtime/character-registry.md` + **如有必要更新** `characters.md` |
| ⏱️ **时间推进** | 本章跨越了多少时间 | `canon.md` 时间线锚点追加行 |
| 📍 **新地点** | 新出现的场景/地点 | 更新 `runtime/character-registry.md` 分章记录 |
| 💰 **财物变动** | 角色收入/支出/资产变化 | 仅记录到 `runtime/character-registry.md`，**不写入** canon.md（除非涉及核心系统规则） |
| ⚙️ **系统揭示** | 本章揭示的新系统规则/限制 | **追加到** `canon.md` 系统规则部分 |
| 🔗 **关系变化** | 本章中出现的重要关系变动 | 更新 `runtime/character-registry.md` 分章记录 |
| 🔮 **新伏笔** | 本章新埋下的伏笔/钩子 | **追加**到 `canon.md` 的 `pending_hooks` 表（状态=🟡） |
| 💖 **情感状态** | 本章角色情感状态的显著变化（恐惧/希望/绝望/动摇等） | 仅记录到 `runtime/character-registry.md`，不写入 canon.md |
| 📋 **物理状态** | 角色物品得失/装备变化/伤势/体力消耗等物理状态变化 | 更新 `runtime/character-registry.md`，影响后续行动能力的物理变化**追加到** `canon.md` |

**执行规则：**

1. **保守更新：** 只有**新揭示的硬事实**才写入 canon.md。角色情绪、个人想法等软信息只记到 `runtime/character-registry.md`
2. **检查确认：** 发现与现有 canon.md 矛盾的事实，不直接改 canon.md，而是按 Step 2.3 的流程处理（先标记矛盾，确认后再更新）
3. **伏笔检测：** 如果本章出现了一个明显会被后续回收的桥段（如「他留下了一个神秘号码」），观察者应主动识别为 🟡 新伏笔并追加到 pending_hooks 表。**不要等用户来告诉你这是伏笔。**
4. **每条新伏笔的格式：**
   ```
   | H{N} | {{伏笔内容}} | ch_{{NN}} 埋设 | ch_{{预期回收}} | — | 🟡 | 由 Observer 自动检测 |
   ```

**输出更新：**
- `foundation/canon.md` — 追加新的时间线锚点、系统规则、pending_hooks
- `runtime/character-registry.md` — 追加本章分章记录
- 如果发现新角色（此前 characters.md 中未记录的），询问用户是否要补充到 characters.md

> **为什么不合并到 Step 2.3？** 因为 Step 2.3 是每 3 章一次的一致性检查（偏重矛盾检测），而观察者是每章一次的事实提取（偏重新事实录入）。分离的两个理由：① 伏笔应在埋下时立即录入，等 3 章可能忘记细节；② 一致性检查较重，观察者很轻，不应绑在一起。

### Step 2.3：章节间一致性检查

每完成 3 章，执行一次一致性检查（参考观察者已提取的 `runtime/character-registry.md` 和 canon.md 的待处理矛盾，不必重读全量章节）：

- 读取 `canon.md`，检查是否有**观察者标记的矛盾**（而非自己通读章节去找）
- 读取 `runtime/character-registry.md`，检查最近 3 章的出场角色是否偏离了 characters.md 中的档案
- 如果有矛盾，更新 `canon.md`（向上反馈）
- 如果矛盾涉及已完成的章节，在 `patches/` 下记录待修复处（Phase 3 处理）
- **字数治理审计：** 读取 `state.json` 最近的 3 章字数记录，检查是否有连续偏短或偏长趋势。如果 3 章中有 2 章标记为 `ok:"short"` 或 `ok:"overshoot"`，在阶段性报告中预警
- **缓存失效更新：** 如果 canon.md 或 voice.md 在本轮检查中被修改，将 `state.json` 的 `canon_stale` 设为 `true`。Composer 在下一章编译时自动读取最新版本

### Step 2.4：阶段性休息提示

每完成 25% 的章节（即写完整部小说的四分之一），暂停并输出阶段性报告：
- 已写章节数 / 总章节数
- 当前平均分
- **字数统计：** 平均每章字数、最长/最短章节、字数治理通过率（🟢占比）
- 节奏评估（有没有拖沓/过快）
- 建议下一步调整方向

## Phase 3：修改循环（Revision）

**目标：** 对已完成的初稿进行多轮修改，提升质量到可发布水平。

### Phase 3a：自动化修改

按以下顺序执行：

#### Step 3a.1：对抗式编辑

##### 模式 A：全本/批量审计（30+ 章小说）

> 使用并行审计策略（见下文），不要单次扫描全本。

##### 模式 B：单章快速审计（用户修改后要求评审）

适用于：用户提交了修改建议、你落地后，用户要求跑对抗式编辑的场景。不走 delegate_task 并行流程，在当前会话直接完成。

**执行步骤：**

1. **阅读 37 维度框架** — 加载 `references/adversarial-review-framework.md`，熟悉所有维度
2. **运行自动化扫描** — 用 `terminal` 或 `execute_code` 运行以下检测：
   ```bash
   # 禁用词扫描（忽然/好像/忍不住/突然/猛地/然而/但是/却）
   grep -n '忽然\|好像\|忍不住\|突然\|猛地\|然而\|但是\|却' <chapter.md>

3. **逐维度评判** — 对 37 维度逐一过评，标注 ✅/⚠️/❌，每个 ⚠️ 或 ❌ 必须附带「位置+诊断+修改建议」

5. **输出待修清单** — 按优先级排列：🔴 硬伤 > 🟡 角色一致性 > 🟢 结构性 > 📌 创造性 > 🔍 细节交付

- 修改建议要可执行——每条建议附带具体的替换文本（before/after）

> **并行审计策略（30+ 章小说推荐）：** 将 37 维度按自然分组拆为 **3 个 delegate_task 并行 worker**，每个负责一组维度，各自读取全部章节后输出该组的审计报告。主 agent 汇总为一篇 editorial-notes.md。
>
> | Worker | 维度组 | 检查重点 |
> |:------|:------|:---------|
> | A | **V1-V8（系统设定）+ V29-V37（细节交付）** | 数据连续性、时代准确、文风合规（加粗/禁用词/感官） |
> | B | **V9-V16（角色一致性）** | 人设稳定性、行为逻辑、配角弧光、角色声音 |
> | C | **V17-V23（结构节奏）+ V24-V28（创造性）** | 三幕比例、伏笔回收、反套路执行、主题一致性 |
>
> **为什么这样分？** ① 每组工作量均衡（A/B/C 各自约 8-9 个维度）；② 维度间关联性强（如 V1 系统规则和 V6 财务连续性都涉及跨章数据核对，放在同一 worker 中不必重复读文件）；③ V29-V37（细节/文风）和 V1-V8（设定/数据）都是需要逐章扫描的密集型检查，合并给 A 最合适。
>
> **每个 worker 的 context 必须包含：** 项目的 path、outline.md 三幕概览、characters.md 核心角色档案、canon.md 完整设定、voice.md 的文风规则（句长/禁用词/加粗限制/段落要求）。**不要给 worker 完整的 outline.md 或所有 chapter 文件列表**——只给大纲摘要和关键规则，让 worker 自行 `read_file` 按需读取章节文件。
>
> **汇总阶段的处理原则：** 发现的问题直接并入 editorial-notes.md 的对应优先级分组。注意去重——可能 A worker 报告了 V1 问题而 C worker 在 V27 中提到了同一问题，只保留一次。已自动修复的 bug 应标记为「已自动修复 ✅」并归入 🟡 级，不列入 🔴。

> **核心原则：** 对抗式编辑不是为了证明自己有多聪明，是为了帮作者（用户）看到自己看不到的问题。每一处批评都必须带「具体位置 + 根因诊断 + 可执行建议」，缺一不可。

#### Step 3a.5：全局角色改名

**适用场景：** 用户确认接受对抗式编辑或角色审查中的改名建议后，需要将改名落地到全项目文件。

**两种模式：**

##### 模式 A：全名替换（姓+名）

如「江美兰→张美静」。全文统一替换，用 sed 即可。

**执行流程：**

1. **备份** — 先备份项目，方便回滚：
   ```bash
   cd ~/novels/<书名>
   tar czf /tmp/novel_backup_$(date +%s).tar.gz .
   ```

2. **sed 批量替换** — 使用 `find + sed -i` 覆盖所有 `.md` 文件：
   ```bash
   # ⚠️ 顺序规则：先长后短，避免部分匹配
   # 例如「江美兰→张美静」必须在「江美舒→张美欣」之前，
   # 否则「江美」前缀先被匹配会导致剩余部分出错
   find . -name "*.md" -type f -exec sed -i 's/江美兰/张美静/g' {} +
   find . -name "*.md" -type f -exec sed -i 's/江美舒/张美欣/g' {} +
   find . -name "*.md" -type f -exec sed -i 's/梁秋润/梁秋实/g' {} +
   ```

3. **验证零残留** — 必须有这一步，不能跳过：
   ```bash
   grep -rn "旧名1\|旧名2" --include="*.md" . || echo "✅ 零残留"
   ```
   如发现残留，检查文件名是否包含旧名（如 `ch_jml.md`），或排查是否在 `state.json`、`runtime/` 中的非 md 文件有残留。

4. **非 md 文件检查** — `state.json`、`manuscript-stats.md` 中的头部引用等非正文文件如果用了旧名，单独处理。

5. **记录到 fix-log** — 在 `revision/fix-log.md` 添加全局改名记录。

6. **更新 revision 文件** — `revision/name-review.md`（如果有）和 `revision/editorial-notes.md` 中的旧名引用在 sed 阶段通常已被自动替换，但需要确认核对。

##### 模式 B：裸称替换（对话/叙述中的无姓称呼）

**适用场景：** 全名已改完（模式 A），但对话和叙述中仍用裸称（无姓）称呼角色。例如「江美舒→张美欣」之后，对话中的「美舒」→「美欣」。

**为什么不能用 sed：** 裸称可能与原文中的名字概念冲突（如「美舒就是美舒，美兰就是美兰」是对仗句，必须保留）。sed 无上下文判断能力。

**执行流程：**

详见 `analyzebook` 技能的 `references/batch-rename-workflow.md` → **方法B：裸称替换、方法C：全局无例外替换**，包含完整 Python 脚本模板、KEEP_PATTERNS 策略、全局替换方案和验证指南。

**关键步骤摘要（详见上述参考文件）：**
1. `grep -rn` 列出所有裸称旧名出现位置
2. 编写 Python 脚本，用 `KEEP_PATTERNS` 保护身份对仗句和名字概念引用
3. 用 `match_pos` 周围 snippet 做位置级判断（而非整行判断——防止同行其他匹配项误保留）
4. 运行后 `read_file` 验证关键上下文
5. 记录到 fix-log.md

**核心保持模式示例：**

```python
KEEP_PATTERNS = [
    r'旧名就是旧名',      # 身份对仗（「美舒就是美舒」）
    r'旧名是旧名',        # 身份对仗（「美兰是美兰」）
    r'甲名和乙名',        # 双胞胎名字概念对（「美舒和美兰」）
    r'说的是旧名',        # 引述旧名字
    r'「旧名」',          # 带引号的名字概念引用
    r'旧名的照片',        # 物品上的名字
]
```

> 📌 **参考：** `analyzebook` 技能 `references/batch-rename-workflow.md` 有完整脚本模板和实战坑列表。

**适用时机：** 可以在 Phase 3 的任意轮次中执行（不限于 Step 3a.4 之后），只要用户确认改名。如果改名发生在 Phase 3a 之后（如深改阶段用户临时决定改名），同样适用上述流程。

#### Step 3b.0：剧情修改确认 🗳️

  1. ch_03 → 女主出场动机不明确 → 补充心理描写

#### Step 3b.1：合并完整手稿

#### Step 3b.2：双人格深度审阅

一次性以「文学评论家 + 写作教授」双人格审阅整个手稿。

**文学评论家视角：** 哪些段落让你读出声来称赞？哪些段落让你走神？
**写作教授视角（实用）：** 逐章给出建设性批评。关注：
- 开场前 3 句话是否有效
- 每个场景是否有明确的目的
- 对话是否自然、是否推进情节或揭示角色

- 情感弧线：读者在每一章体验到的情绪曲线

#### Step 3b.2b：字数治理修复（InkOS 融合）

> 在深改之前，先治理字数偏差。原因是：字数偏差会干扰深改的判断——长章会让审阅者觉得「拖沓」，短章会觉得「单薄」，先修到一个合理范围再审。

**执行流程：**

1. **扫描字节：** 遍历所有 `chapters/ch_*.md`，统计每章 `zh_chars`（去空白中文字符数，不含标点/英文/数字）
2. **对照 state.json：** 读取 `word_governance.per_chapter` 中的 target 和 actual
3. **偏差分类：**

| 标记 | 含义 | 处理 |
|:----|:-----|:-----|
| 🔴 **< 50% target** | 严重不足 | 写入修复队列（优先级最高） |
| 🟡 **short** | 偏少（50-70%） | 写入修复队列（中优先级） |
| 🟢 **ok** | 合格 | 跳过 |
| 🟡 **overshoot** | 超长（130-200%） | 写入修复队列（中优先级） |
| 🔴 **> 200% target** | 严重超长 | 写入修复队列（最高优先级） |

4. **修复策略：**

| 问题 | 修复方式 | 示例 |
|:----|:---------|:-----|
| 偏短 | ① 扩展关键场景（加感官细节）② 增加角色内心戏 ③ 补充环境氛围 ④ 不要为了凑字数加无效词句 | 原：`梁秋润没说话。` → 扩展：`梁秋润没说话，端着茶杯的手停在半空，杯沿的水汽模糊了他的表情。` |
| 偏长 | ① 砍掉冗余的对话回复 ② 合并连续短段 ③ 删除不推动剧情的环境描写 ④ 检查是否混入了不同 POV 的内容（见违规案例） | 「好的，我明白了，那我先回办公室了。」→「那我先走了。」|
| 严重超长 | ① 尝试拆分到下一章 ② 或大幅压缩 20-40% | 6000字章拆为 3000+3000 两章 |

5. **输出治理报告：** 包含每章治理前后的字数对比，保存到 `revision/word-governance-report.md`
6. **更新 state.json：** 所有修复后，将 `actual` 更新为修后的字数，`ok` 更新为 `true`

> **警示：** 字数治理的目的是让所有章节在**合理的篇幅范围内**，不是让所有章节字数一模一样。±20% 的差异是完全正常的——只要不是「300字 vs 8000字」那种极端情况。过度追求字数统一会杀死作品的节奏感。

#### Step 3b.3：逐个修复循环

1. 按优先级顺序读取 deep-review.md 中的问题
2. 对每个问题进行修复
3. 记录到 `revision/fix-log.md`（格式见下方）
4. **连续执行模式：** 修复过程中发现的新问题，**不要停下询问用户**，直接记录到 fix-log 的「新增发现」区，继续当前修复。一条条修完后，再统一向用户汇报新发现并确认是否纳入下一轮。用户明确要求「不要停发现新问题也一起改」时，必须采用此模式。
5. 全部修复后，**回到 Step 3b.1 重新合并，进入新一轮深度审阅**
6. 当一轮深度审阅中无明显缺陷（标注为「无需修改」的问题 ≥ 90%）时，退出循环

> **注意：** 这个循环可能进行 2-3 轮。不要在这里追求完美——过度修改会杀死作品的生动性。目标是「无明显缺陷」而非「无可挑剔」。

**修复日志（fix-log.md）标准格式：**

```markdown
# 修复日志 · 《小说名》

## 第N轮修改

### ✅ P1 · 问题标题（日期）

**文件**：`chapters/ch_NN.md`（新建/修改）
**操作**：在XX章和XX章之间插入/修改了什么
**Before**：旧设定/旧剧情一句话
**After**：新设定/新剧情一句话
**状态**：✅ 完成

### 📌 新增发现（连续执行模式下记录）
| # | 问题 | 位置 | 发现时间 |
|:-:|:----|:----|:---------|
| P8 | 系统规则不一致 | 第4章vs第24章 | 修复P1过程中 |
```

## 短篇模式（Short Story Mode）⚡ *InkOS Short Run 融合优化版*

> 独立短篇不走完整 4-Phase 长篇管线——太慢了。
> 短篇模式是轻量版：快速建角色→一次写完→轻审→导出包含简介卖点+封面提示词。

**何时使用：** 3-12 章、每章 500-2000 字的独立短篇。超过这个量建议走完整长篇管线。

**与完整管线的核心差异：**

| 维度 | 长篇管线 | 短篇模式 |
|:-----|:--------|:---------|
| Foundation 文件 | 5件套 + 评分门控 | 仅 characters.md + outline.md |
| 每章流程 | Plan → Compose → Draft | 一次 delegate_task 批量写完 |
| 审计 | 37维度 + 读者评审 | 5维度轻量检查 |
| 修改循环 | 多轮 | 最多1轮 |
| 导出 | manuscript.md | manuscript.md + sales-package.md + cover-prompt.md |
| 总 Token 消耗 | 高 | 低（约1/5） |

### 执行流程

#### Step S0：输入方向

由用户提供一句话 premise。例如：
> "都市婚姻反转短篇：女主结婚三年发现老公和小三密谋转移财产，她提前拿到账本证据，在离婚协议上反杀。"

如果用户输入不完整，引导补全以下信息：
- **故事方向**（1-3句话）
- **章节数**（默认 6 章）
- **每章字数**（默认 1200 字）

#### Step S1：快速地基

创建轻量 foundation，只写**必要的内容**：

**`foundation/characters.md`**（仅核心角色，不要求深层需求/弧光/谎言等）：

```markdown
# 角色档案（短篇）

## 主角：林晚
- 28岁，全职主妇，结婚三年
- 表面温顺，内有乾坤。算账型人格
- 发现老公转移财产后，冷静收集证据，准备反击

## 配角：赵铭
- 32岁，创业公司CEO，林晚的老公
- 自以为掌控一切，实则核心资产全靠老婆娘家早期投资
- 出轨对象：公司财务刘心怡

## 配角：陈律师
- 林晚的大学同学，离婚律师
- 专业、冷静，帮林晚设计财产保全方案
```

**`foundation/outline.md`**（每章1-2行，不写悬念/钩子/节奏等）：

```markdown
# 章节大纲（短篇）

总章节数：6章 | 每章字数：约1200字

第1章：发现端倪 — 林晚整理家务时发现银行流水异常
第2章：暗中查证 — 跟踪赵铭，确认出轨+转移资产
第3章：收集证据 — 找陈律师，设计财产保全方案
第4章：摊牌 — 林晚主动约赵铭谈话，出示证据
第5章：离婚协议 — 谈判桌上的心理博弈，林晚反杀
第6章：新生 — 离婚后的林晚，开自己的工作室
```

> **不做 world.md / voice.md / canon.md**——短篇信息量小，直接放在 outline 和 characters 里就够了。
>
> **不做评分门控**——短篇追求效率，地基差不多就开写。

#### Step S2：批量撰写

**一次 delegate_task 写完所有章节**（不逐章 Plan→Compose→Draft）：

```markdown
子任务 instructions:
1. 读取 characters.md 和 outline.md
2. 一次性写完第1章到第{ N }章
3. 每章保存为 chapters/ch_NN.md
4. 每章完成后做字数检查（目标±30%）
5. 遵循禁用词：不出现「突然」「好像」「仿佛」「忽然」「忍不住」「猛地」「却」
6. 不写大段环境描写，每段不超过5句
7. 对话标记用行动式，极少「XX说」
```

**字数治理规则（短篇宽松版）：**

| 偏差 | 处理 |
|:----|:-----|
| < 50% target | 🔴 严重不足，补写到 ≥ 60% |
| 50-130% target | 🟢 合格 |
| > 130% target | 🟡 超长，标记留待审阅决定 |

> **为什么不做逐章 Plan→Compose？** 短篇每章字数少（500-2000字），Plan→Compose 的 overhead（每章读 intent→读 outline→读 voice→编译 context→写）比写正文本身还重。用一次 delegate_task 把全篇写完，上下文一致性好，还省 token。

#### Step S3：轻量审阅

**5维度快速检查，不跑 37 维审计，不跑读者评审：**

| 维度 | 检查内容 |
|:-----|:---------|
| 情节逻辑 | 故事有没有硬伤？起承转合完整吗？ |
| 角色一致性 | 角色行为是否符合出场设定？ |
| 文笔合规 | 禁用词零命中？对话自然？节奏不拖？ |
| 悬念/钩子 | 开头抓人吗？结尾有回味吗？ |
| 整体完成度 | 字数达标？结尾收住了吗？ |

**格式：**

```markdown
# 轻量审阅报告 · 《短篇名》

| 维度 | 评分 | 问题 |
|:-----|:---:|:-----|
| 情节逻辑 | 8/10 | 第4章摊牌场景赵铭反应不够真实 |
| 角色一致性 | 7/10 | 林晚在第2章的行为偏被动，与后期人设不符 |
| 文笔合规 | 9/10 | 禁用词零命中 |
| 悬念/钩子 | 8/10 | 开头有力，结尾还可以更利落 |
| 整体完成度 | 8/10 | 字数达标，情绪弧线完整 |

**修改建议：**
- P1 第4章赵铭的反应加一句心理破防描写
- P2 第2章林晚查账时加一个主动决策时刻
- P3 第6章结尾砍掉最后一段抒情
```

**修改执行：** 按优先级逐一修复。最多 1 轮，不循环。

#### Step S4：导出三件套

##### 1. 合并手稿

```bash
cat chapters/ch_*.md > manuscript.md
```

##### 2. 生成简介卖点（sales-package.md）

```markdown
# 简介卖点 · 《短篇名》

## 一句话梗概
{用一句话说清楚核心冲突和反转}

## 完整简介（200-300字）
{适合发布到平台的简介正文}

## 卖点标签
{3-5个标签，如：婚姻反转|爽文|女主觉醒|短篇}

## 适合平台
- 小红书：{适配小红书风格的推荐语}
- 抖音：{适配抖音口播风格的推荐语}
- 番茄/晋江：{适配网文平台的推荐语}
```

##### 3. 生成封面提示词（cover-prompt.md）

```markdown
# 封面提示词 · 《短篇名》

{A detailed prompt for AI image generation}

## 中文描述
{简洁的中文画面描述}

## 推荐风格
{都市/古风/悬疑/言情/...}

> **为什么多这两个文件？** 短篇的核心场景是发布（小红书/抖音/番茄），不是存档。sales-package.md 和 cover-prompt.md 是发布时的刚需——写完整篇之后最头疼的两件事（怎么写简介、配什么图），一次生成。

## 常见陷阱

### 🚫 陷阱 1：地基不够就开写
**表现：** 跳过或不认真执行 Phase 1 的门控评分，直接进入初稿。
**后果：** 写到一半发现设定矛盾/角色平庸/大纲撑不起长篇，需要大改甚至重来。
**解决：** 严格执行 foundation_score ≥ 7.5 门控。如果 3 轮修改后仍不够，诚实地告诉用户这个种子概念可能不成熟。

### 🚫 陷阱 2：评分变成走过场
**表现：** 所有章节都打 6.0+，从不触发重写。
**后果：** 质量低劣的章节累积到 Phase 3 才暴露，修复成本远高于当场重写。
**解决：** 自评时问自己一个关键问题：「如果这是一本你花钱买的书，读到这一章你会不会觉得被坑了？」诚实回答。

### 🚫 陷阱 3：过度修改（over-polishing）
**表现：** 在 Phase 3b 中无限循环，对每个标点符号吹毛求疵。
**后果：** 产出"打磨到死"但失去生气的作品；或干脆永远完不成。
**解决：** 设定退出条件：「无明显缺陷条目 ≥ 90%」。剩下的 10% 可能是个人偏好问题，留给实际读者判断。

### 🚫 陷阱 4：忽视向上反馈
**表现：** 写作中发现设定矛盾，不更新 canon.md 或 world.md。
**后果：** 越往后写矛盾越多，最终 canon 文件失去参考价值。
**解决：** 每完成 3 章强制执行一致性检查。矛盾必须同时修复已写章节和更新 canon。

### 🚫 陷阱 5：声音漂移
**表现：** 第一章和最后一章的文风差异大到像两个人写的。
**后果：** 作品缺乏整体感，读者出戏。
**解决：** 写每章前快速重读 voice.md 的「量化风格指纹」部分（尤其是句长分布目标和词汇禁忌）。如果发现漂移，在 Phase 3a 的统一编辑中纠正——用 voice.md 的基准数据做客观对照，不靠感觉判断。

### 🚫 陷阱 6：章节太短或太长
**表现：** 有的章节 300 字，有的章节 8000 字。
**后果：** 节奏失控，读者体验割裂。
**解决：** 设定每章目标字数范围，参考 `references/word-governance.md` 的场景类型建议。超出范围的在章节自评阶段标记，并写入 `state.json.word_governance`。

### 🚫 陷阱 8：state.json 与实际文件系统不同步

**表现：** 项目经过多轮修改后，state.json 的 `word_governance.per_chapter` 中 `actual` 为 null、`ok` 为 null，或 `completed_chapters` 不完整；revision-brief 仍显示 ⬜ 但实际已修复。

**后果：** 下一轮工作时数据错误，字数治理审计失效，用户需要重新告知进度。

**解决：** 每次 session resume 时（见「续写模式」），用 Python 遍历所有章节文件重新统计真实中文字数，与 state.json 对比，不一致时以文件系统为准更新 state.json。不要在 terminal 中用 shell+grep 做字数统计——`execute_code`（Python re.findall）更稳定。terminal 中复杂的 Unicode grep 在大文件集上容易超时（blocked）。

### 🚫 陷阱 9：delegate_task 超时误判为失败

**表现：** 用 `delegate_task` 分发章节写作，任务返回 `status: "timeout"`，以为文件没写成，于是重试——浪费了子任务的工作。

**真相：** delegate_task 的 600 秒超时通常发生在**汇报阶段**（子任务写完了文件但在整理报告时被截断）。**文件已经安全写入磁盘。** 收到超时后，用 `ls chapters/` 确认新文件是否已创建。如果文件存在（通常如此），不要重试。

**解决：** 将任务拆小（每批 ≤10 章）是避免超时的最好方法。如果必须大批次且超时频繁，信任文件系统的结果而非 API 返回的 status。

### 🚫 陷阱 10：忘记更新 manuscript-stats.md

**表现：** 项目运行多轮后 manuscript-stats.md 仍显示「约XX万字（待填充）」、pass rate 过时、或文件清单与实际不匹配。

**后果：** 用户查看时以为项目未完成，增加不必要的沟通成本。

**解决：** 每次 Phase 边界或 session resume 时，重新计算字数统计并更新 manuscript-stats.md。不要留占位符到「以后填」——改完就填。质量指标部分应包含对抗式编辑修复数、角色名审查数等 Phase 3 成果。

### 🚫 陷阱 7：角色行为不一致
**后果：** 角色失真，读者失去代入感。
**解决：** 写每章前重读 characters.md 中出场角色的档案。在 Phase 3a 的对抗式编辑中重点检查。

### 🚫 陷阱 12：章节重排后旧版「僵尸文本」残留

**表现：** 在已有章节序列中插入新的前置章节（如重写第1章、新增序章）后，后续章节中仍保留着与新前提冲突的旧版描述——例如 ch_02 仍展示旧版招募签约场景，与新版 ch_01 的内容完全矛盾。

**后果：** 读者在新旧版之间感到断裂——ch_01 已经讲了冯枫怎么签约、怎么拿到手环，ch_02 又从头讲一遍不一样的版本。最坏情况：同一角色有两种不同版本的起源故事。

**根因诊断：** 章节重排时，旧 ch_01 被复制为 ch_02，但它的开篇部分引用了旧版前提（招募、签约、能力选择），这些引用在新版 ch_01 中已经被替代或修改了。**文件被复制了，但前提被更新了——复制体成了僵尸。**

**解决：**

1. **rename/copy 后必须检查「旧版前提相关片段」**。重点检查的字段：
   - 角色登场方式（新版 vs 旧版的差异）
   - 装备描述（如「黑色金属手环」→「皮下光纹」、「注射式植入」→「纳米节点」）
   - 能力获取方式（新版是「公司匹配推荐」vs 旧版是「菜单选择」）
   - 招募/签约场景（如果有，必须剪掉——这部分已属于新版前置章节）
   
2. **标准的连续性检查清单：**
   ```python
   # 在章节重排后运行
   old_refs = ['黑色手环', '金属手环', '灰色制服', '菜单', '选择能力', 
               '技能库', '被抽中了', '金属桌子']
   for ref in old_refs:
       if ref in content:
           print(f'⚠️ 旧版引用残留: {ref}')
   ```

3. **过渡句替代方案：** 剪掉旧版场景后，不能直接留空白。用 1-3 行过渡句替代：
   > 「三个小时前还在签合同，三个小时后被空投到雨林里」
   > 这一句就把 ch_01 的场景结论带进 ch_02 的起点，不需要重新展示签约。

4. **小纲也必须跟进：** 小纲_第N章.txt 如果引用了旧版事件顺序（如「签约→园区→……」），在章节重排后需要更新章节编号和事件引用。

### 🚫 陷阱 13：第一个词就踩禁词

**表现：** 整章写得流畅自然，但第一句/第一个场景中就出现了禁用词（如「站台突然变了味」中的「突然」）——这是在 draft 阶段最容易被漏掉的违规类型，因为开头句往往写得很顺手，不会主动触发审稿意识。

**根因：** 写手 agent 在生成正文时专注于场景构建和节奏，容易把「突然」「猛地」「仿佛」等看似自然的起笔词混进开篇。这些词在阅读时不易察觉问题，但会累积到 Phase 3 才暴露。

**解决：**
1. **draft 完成后立即扫描** — 在自评步骤（Step 2.2 第8步门控判断）之前，先跑一次禁用词扫描，而不是等对抗式编辑再发现。实测脚本：

```python
import re
path = 'chapters/ch_NN.md'
with open(path) as f:
    c = f.read()
forbidden = ['忽然','好像','忍不住','突然','猛地','然而','但是','却']
hits = {w: c.count(w) for w in forbidden if w in c}
emotions = {w: c.count(w) for w in ['愤怒','悲伤','恐惧','害怕'] if w in c}
old_refs = {}  # 根据项目实际情况填入旧版遗骸词
print(f"禁用词: {hits if hits else '✅'}")
```

2. **重点检查前100字** — 禁用词最容易出现在章节开头（开场白、过渡句、第一句动作描写）和段落结尾（转折连词「然而」「但是」「却」）。前 100 字和后 100 字是高发区，优先排查。

3. **零容忍** — 发现即修。常用替换：「突然」→ 删除或改为「就在这时候」；「猛地」→ 删除；「忍不住」→「没忍住」或重写为动作；「却」→ 用断句替代（句号断开比转折连接更符合动作节奏）。

### 🚫 陷阱 14：连续 patch 编辑导致上下文损坏

**表现：** 在一章内发现多处问题（禁用词、情绪词、旧版遗骸），用 `patch` 工具依次修复。第二次或第三次 patch 时，虽然返回了成功的 diff，但实际替换的内容与预期不符——旧字符串匹配到了错误的行，导致上下文出现重复行或语法断裂。

**根因：** `patch` 使用模糊匹配（9 种策略）。当一个文件中存在多个相似的文本片段（例如同一角色名出现多次，或同一句式结构在多个段落出现），后续 patch 的 old_string 可能匹配到已被前一次 patch 修改过的文本，而不是目标位置。

**解决：**
1. **每轮只修一个文件中的一个问题** — 执行 `patch` 后立即 `read_file` 验证该处修改结果，确认正确后再执行下一个。不要在一个 turn 中发送3-5个连续 patch。
2. **加入足够上下文** — old_string 包含目标行 + 前后各1行（共3行），确保唯一性。仅靠单独一行文本匹配容易误匹配。
3. **发现异常后检查全文件** — 如果某次 patch 的 diff 看起来比预期长或短，立即 read_file 检查该区域，不要假设成功。
4. **重构方案** — 如果一章内需要修复超过3处，优先考虑替换策略：直接用 `execute_code` 编写 Python 脚本批量修复（正则匹配+替换），而非逐个 patch。

**实战案例（2026-07-07）：** 测试 plan→compose→draft 管线时，ch_01.md 开头有「站台突然变了味」。第一轮 patch 替换了「突然」但错误地同时将下文的「周小姐看着他的眼睛」对话也替换掉了，导致对话台词丢失。需要用第二轮 patch 找回丢失文本。第三次修复时改用「前+后各1行」的 old_string 格式，一次性正确修复。

### 🚫 陷阱 14：文件属主错误（root vs 用户）

**表现：** foundation/、chapters/、runtime/ 等目录下的文件 owner 为 root，用户在宿主机无法直接编辑。

**根因：** Hermes Agent 在 root 环境下运行，write_file 创建的文件 owner 是 root。

**解决：** 在 Phase 2 写完一批章节后或 Phase 4 导出前执行一次：
```bash
chown -R gemmt:Users /vol1/1000/data/ai文案/novels/<项目名>/
```
`chown` 命令在此环境中可用。确认项目路径存在后执行即可。也可以在 Step 2.1（准备阶段）直接加入 chown 步骤预防此问题。

### 🚫 陷阱 11：项目目录用了章节名而非书名

**表现：** 创建小说项目目录时，直接用了第一幕/第一章的标题（如「营救苏允儿」）作为 `~/novels/` 下的目录名，而非正式书名（如「末日降临，末世游戏」）。

**后果：** 后续需要重命名目录、修改所有文件中的书名引用、更新 state.json 的 novel_name 字段——增加不必要的操作成本。

**解决：**
1. **创建目录前，先问用户书名。** 如果用户只提供了章节名而非书名，在创建 `~/novels/<name>/` 之前明确询问：「这本小说的正式书名是什么？」
2. 如果用户说「先用这个名字」，在 state.json 备注字段标注 `"provisional_title": true`，后续确认正式书名后再更正
3. 书名中包含特殊字符（如书名号、冒号、逗号）时，用下划线替代——`~/novels/末日降临_末世游戏/` 而非 `~/novels/《末日降临，末世游戏》/`
4. 如果已经建错了，修正流程：先改 foundation/*.md 和 state.json 中的书名引用 → 重命名目录 → 更新文件名（如 `大纲_旧书名.txt` → `大纲_新书名.txt`）→ grep 零残留验证
